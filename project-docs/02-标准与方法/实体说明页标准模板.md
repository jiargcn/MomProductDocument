# 实体说明页标准模板

> 状态：2026-07-16 重构。适用于主数据、配置、业务单据、任务、记录、查询和公共能力页面。
> 首要目标：让业务人员能看懂、培训人员能讲清、使用者能快速找到规则；不是把源码或数据库逐段翻译成页面正文。

## 1. 写作分层与边界

| 层次 | 放置位置 | 回答的问题 | 不应出现的内容 |
| --- | --- | --- | --- |
| 业务说明 | `docs/` 正式页面正文 | 为什么要做、谁来做、如何完成、做完有什么影响。 | DDL、DO、DTO、Controller、源码路径等取证过程。 |
| 页面字典 | `docs/` 正式页面后半部分 | 如何新增、编辑、查询、导入、联查。 | 低价值的全字段技术罗列。 |
| 证据与底稿 | `project-docs/` | 结论依据是什么、字段技术名如何校正、存在什么冲突。 | 直接作为培训结论发布。 |

正式页面只在页首用一行写明“适用版本/更新日期”。“代码已证实”“DDL/DO 已证实”“待核验”等取证标签不进入业务叙事；必要时在末尾的“产品观察”中用自然语言说明限制。

## 2. 先建立完整大纲，再填内容

任何页面第一次建立时，必须一次性列出与该页面类型相适应的完整章节。暂时没有资料时，保留下面这种**可读的站位段落**，不以空表或技术术语代替。

```markdown
## 业务目的

【待补充：用一到两段说明该页面解决什么现场或管理问题，谁在什么场景下使用，以及不用它会有什么后果。】

## 业务流程

【图示占位：一张“触发 → 处理 → 结果”的流程图。复杂单据应包含来源、申请、任务、记录和下游影响。】

【待补充：结合一笔典型业务，按操作顺序解释每一步。】

## 常见异常与处理

【待补充：至少列出取消、数量差异、无权限、数据不满足条件等与本业务相关的情形。】

【截图占位：异常提示或处理入口的页面截图，需脱敏。】
```

站位段落必须回答“后续要写什么、为什么要放图/截图、需要什么资料”，但不得写成“待 DDL/DO 核验”或只留一个空标题。

## 3. 正式页面的通用结构

按页面类型选择章节；不适用的章节写明“不适用及原因”，不要机械堆表。

1. **业务目的与适用范围**：解决的问题、使用角色、开始与结束边界。
2. **使用前准备**：前置主数据、配置、权限或上游单据。
3. **典型业务流程**：以一笔具体业务说明操作顺序和每一步结果。
4. **关键规则与状态**：什么条件能做什么，状态如何变化，失败会怎样。
5. **页面操作说明**：新增、编辑、提交、执行、关闭等用户真正会做的操作。
6. **查询与追溯**：列表应看什么、详情如何分组、可联查哪些对象。
7. **相关业务与影响**：上游、下游、库存、质量、设备、接口、报表等影响。
8. **常见异常与处理**：常见问题、处理入口、需找谁协助。
9. **图示、截图与示例**：按需放流程图、状态图、页面截图和脱敏示例。
10. **产品观察**：已确认但会影响使用的限制、待业务确认项及其影响。

## 4. 主数据与配置页的页面字典

先按“维护任务”讲清楚，再提供少量高价值字段信息。

### 4.1 建议章节

1. 这项数据是什么，被哪些业务使用。
2. 谁负责维护，新增前要准备什么。
3. 如何新增：用户需要填写什么、从哪里选择、系统自动带入什么。
4. 如何修改、启停、删除：何时允许，影响什么。
5. 如何导入：模板应填什么，不应填什么，常见错误如何处理。
6. 如何查询：默认应显示哪些字段、常用筛选条件、详情如何查看。
7. 典型示例与常见错误。

### 4.2 推荐表格

| 表格 | 推荐列 | 用途 |
| --- | --- | --- |
| 关键维护信息 | 业务字段、含义、谁填写、何时填写、填写规则 | 只列影响识别、维护和下游使用的字段。 |
| 选择与带入 | 要选择什么、可选范围、带回什么、限制原因 | 解释选择器，而不是罗列接口参数。 |
| 导入模板 | 模板列、用户填写/系统填充、校验、错误处理 | 直接服务导入操作。 |
| 列表与详情 | 列表顺序、详情分组、联查对象 | 服务查询与培训。 |

类型、长度、技术字段名只在确实会影响用户录入或导入时，在“系统限制说明”小表中按需列出；完整映射留在工作底稿。

## 5. 业务单据、申请、任务与记录页

页面首先按一条业务链讲故事。公共的“申请—任务—记录”概念只引用 [申请、任务与记录模型](../../docs/02-业务模型/01-申请任务记录模型.md)，本页只写该业务的差异。

### 5.1 建议章节

1. 触发场景：谁因为什么业务事件发起。
2. 一笔业务怎么走：来源单据 → 申请 → 任务 → 现场执行 → 记录 → 下游结果。
3. 各角色分别做什么：发起、审核、承接、执行、复核、查询。
4. 状态与动作：以业务语言说明“什么状态下可做什么”。
5. 库存、质量、设备或接口影响：说明业务结果，不先展示服务调用。
6. 列表、详情与联查：使用者如何找到来源、任务、记录和结果。
7. 正常与异常示例：至少一笔正常、一笔常见异常。

### 5.2 站位示例：采购收货

```markdown
## 一笔采购收货如何完成

【待补充：以“供应商送达 100 件物料”为例，说明从采购订单/送货通知到收货完成的操作顺序、参与角色和结果。】

【图示占位：采购订单/ASN → 收货申请 → 收货任务 → PDA/WEB 收货 → 收货记录 → 上架或检验 → 库存可用的流程图。】

## 收货后对库存和相关单据的影响

【待补充：说明收货后会产生哪些待处理或实际库存结果，以及后续可进入上架、检验、退货等哪些业务。】

【示例数据占位：采购订单 PO-xxx，物料 A，计划 100，实收 98、拒收 2；展示每一步看到的数量变化。】
```

## 6. 查询、报表与公共能力页

以“使用者如何完成判断或操作”为中心：数据从哪里来、何时更新、看哪些指标/字段、如何筛选、何时导出、能联查什么、查不到或打印失败时怎么办。系统配置、接口路径、日志对象等只写到足以让读者操作或定位问题的程度。

## 7. 图示、截图与示例的最低要求

| 内容 | 何时需要 | 站位写法 |
| --- | --- | --- |
| 流程图 | 多角色、多单据或跨模块流转。 | 【图示占位：说明起点、终点、关键分支和要表达的关系。】 |
| 状态图 | 动作受状态限制。 | 【图示占位：列明状态、允许动作和回退/关闭分支。】 |
| 页面截图 | 需要解释列表、表单分组、按钮或异常提示。 | 【截图占位：说明页面、区域、脱敏要求和要让读者看懂的内容。】 |
| 示例数据 | 数量、状态、关联关系或导入规则较复杂。 | 【示例数据占位：说明场景、关键数据和预期结果。】 |

## 8. 发布前自检

- 开头三屏内能否说清“这是什么、谁来用、怎么完成一次典型业务”？
- 去掉技术字段名和证据标记后，页面是否仍可读、可讲？
- 占位是否明确写出后续内容、图示/截图类型和所需资料？
- 是否把完整字段映射、源码路径、表名和证据冲突留在 `project-docs/`？
- 是否至少有一张流程/状态/关系图或明确的图示占位？
- 产品限制是否集中在末尾，而没有打断主流程？

## 9. 禁止事项

- 不把 DDL、DO、DTO、VO、Controller、接口调用链当作业务正文。
- 不以“当前实现事实卡”“代码已证实”等技术标签开篇。
- 不用空表代替站位段落。
- 不为了覆盖字段而打断一条完整业务流程。
- 不把未确认的实现细节、历史规划或产品设想写成用户可依赖的规则。
