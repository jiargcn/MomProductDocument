# 字段技术名真实性校正任务清单

> **【归档文件】** 过程记录，默认不必日常关注。日常入口见 [`00-需关注`](../../00-需关注/README.md)。


> 优先级：P0。  
> 执行时点：模块入口页完成后、叶页字段字典和导入规则大规模修订前。  
> 基线：测试环境 / `dev` 分支 / 2026-07-15。

## 1. 问题与整改目标

现有文档中存在“根据前端中文标签自行翻译出英文字段名”的历史问题，导致文档字段名可能与后端 DTO、DO 和数据库真实列不一致。

本任务的目标不是把文档变成数据库表说明书，而是保证每一个写入文档的技术字段名都可追溯、可验证：

- 中文名称用于培训和业务阅读；
- 技术名称用于查询、接口联调、导入、测试和问题定位；
- 两者必须成对出现，且不得通过中文标签反向猜测技术名称。

## 2. 证据优先级与判定规则

| 优先级 | 证据 | 可确认内容 | 使用规则 |
| --- | --- | --- | --- |
| 1 | 数据库 DDL / 实际表结构 | 表名、列名、数据库类型、长度、默认值、索引/约束 | 数据库存储字段的唯一权威来源。 |
| 2 | 后端 DO / Entity / Mapper | Java 属性、列映射、逻辑删除、租户/审计字段 | 用于确认后端属性与数据库列的映射。 |
| 3 | API DTO / VO / 接口注解 | 请求字段、响应字段、校验、接口语义 | 用于确认前后端交互字段，不等同于数据库列。 |
| 4 | 前端 TypeScript 类型、表单、列表与请求参数 | 前端字段键、展示/编辑行为、选择器和默认值 | 用于确认 UI 使用的字段键和中文标签。 |
| 5 | 页面中文标签、历史文档、旧规划书 | 业务名称和历史语义 | 只能解释含义，不能单独推导技术字段名。 |

发生冲突时，不以“看起来更合理”的名称替代真实字段；应标记为“待确认”，并记录冲突来源。

## 3. 标准字段映射表

每个业务页面的字段校正都先生成以下映射表，再回写产品文档：

| 业务中文名 | 前端字段键 | API DTO 字段 | 后端 DO 属性 | 数据库表.列 | 类型/长度 | 文档结论 | 证据位置 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 示例：回收件 | `isRecycled` | `isRecycled` | `isRecycled` | `basic_itembasic.is_recycled` | 待核验 | 已确认/待确认/冲突 | 文件路径与类/表名 |

说明：同一业务字段在“页面 DTO”“运行表”“历史表”中可能对应不同技术字段；不得强行合并。文档应按对象说明其差异。

## 5. 执行顺序

1. 先扫描全站既有字段表，建立 FTA-001 问题清单，不直接批量替换。
2. 按模块建立技术字段证据索引；优先使用当前 `dev` 的 DDL、DO、DTO。
3. 以既有 WMS 纵向试点完成 FTA-003，验证映射表、冲突处理和文档写法。
4. 依次处理 DBC → WMS → MES → QMS → EAM → SCP → ANDON → System/Infra。
5. 每个模块完成后运行链接/构建校验，并抽样由开发或业务人员复核高风险字段。

## 6. 特殊处理规则

| 场景 | 处理规则 |
| --- | --- |
| 前端字段键与数据库列不同 | 两者都记录，明确前端键、DTO 属性和数据库列的对应关系。 |
| DTO 包含计算/显示字段 | 标记为“接口/展示字段”，不得当作数据库字段或导入列。 |
| 数据库审计字段出现在 Excel VO | 除非导入代码证实可写，否则默认列为系统填充，不进入业务导入模板。 |
| 历史文档字段无法找到证据 | 将技术字段名改为“待核验”或移除，仅保留经确认的中文业务含义。 |
| 字段语义冲突 | 保留冲突事实，进入产品差距总账；例如已确认的 `isRecycled` 中文语义为“回收件”。 |
| 多表/主子表页面 | 按主表、明细表、运行表、日志表分别列出，禁止把字段混写为同一对象。 |

## 7. 验收标准

- 所有已修订页面中的技术字段名都有 DDL、DO、DTO 或前端类型证据。
- 不再存在仅由中文标签推导出的英文字段名。
- 字段表能区分“数据库列”“后端属性”“API 字段”“前端字段键”和“展示名称”。
- 导入、选择器、列表和详情说明使用的是对应层级的真实字段。
- 任何冲突或缺失都有编号、证据和待确认责任，不被静默掩盖。
