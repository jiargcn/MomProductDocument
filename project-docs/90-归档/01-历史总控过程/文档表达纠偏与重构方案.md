# 文档表达纠偏与重构方案

> **【归档文件】** 过程记录，默认不必日常关注。日常入口见 [`00-需关注`](../../00-需关注/README.md)。


> 发起原因：2026-07-16 对 P0 样板与批量占位页抽查发现，正式页面过早吸收 DDL、代码取证、字段映射与差距登记，读起来像研发底稿，未达到“培训材料 + 业务字典”的目标。

## 1. 已确认的问题

| 问题 | 当前表现 | 影响 |
| --- | --- | --- |
| 站位不是大纲 | 许多页面以“待核验表格”或技术章节起步。 | 无法一眼看懂页面最终要讲什么，也无法分派业务、图示、截图任务。 |
| 业务叙事被取证打断 | “DDL/DO 已证实”“代码已证实”频繁出现。 | 培训对象需要先理解技术实现，业务逻辑反而不突出。 |
| 字段表过早、过宽 | 技术字段、类型、层级和接口细节进入主正文。 | 查询成本高，且容易把技术实现误读为业务规则。 |
| 差距散落正文 | 取证冲突和产品问题混在流程描述中。 | 用户难以判断什么能做、什么只是内部观察。 |

这不是取证本身错误，而是**取证底稿和正式页面混层**。保留已有证据成果，但调整其呈现位置和后续工作顺序。

## 2. 新的交付标准

正式 `docs/` 页面必须优先满足以下阅读路径：

```text
我为什么需要它 → 我需要先准备什么 → 我如何完成一次业务 →
什么情况下能做什么 → 做完会影响什么 → 如何查询和处理异常
```

页面字典只保留用户做判断所需的字段、规则、导入和查询信息；真实技术字段、类型长度、接口/代码证据、冲突比对等，转入 `project-docs/` 的底稿或任务记录。必要的技术限制可在正文末尾用“系统限制说明”简述。

## 3. 站位段落的统一定义

每个新建或重构页面，先一次性建立完整业务大纲。每个未填章节至少包含：

1. `【待补充】`：将要解释的业务问题和目标读者；
2. `【图示占位】`：所需图的类型、要表达的关系；
3. `【截图占位】`或`【示例数据占位】`：需要什么页面/数据、用于说明什么。

这类站位是可分派的内容任务，不是“资料不足”的技术备注。完整标准见[实体说明页标准模板](../00-需关注/实体说明页标准模板.md)。

## 5. 原有成果如何处理

| 已有内容 | 处理方式 |
| --- | --- |
| 流程图、状态图、领域模型关系 | 保留并优先放到业务流程附近。 |
| 已确认的业务事实 | 改写为自然语言结论，保留在正式页。 |
| DDL/DO/DTO/VO、源码调用链、精确字段映射 | 移至或保留在 `project-docs/` 证据底稿。 |
| 产品差距与前后端冲突 | 正式页末尾用简短“产品观察”说明；完整编号、证据和跟踪留内部总账。 |
| 空表、待核验表、技术化站位 | 替换为完整章节 + `【待补充/图示/截图/示例】` 站位。 |

## 6. 继续推进的闸门

在 DBC/WMS 完成第一阶段准入前，不恢复 BATCH-02 的批量技术回填。P0 样板已完成文档侧复审，但仍需与模块首页、全量分组大纲一并进行最终确认。确认标准：

- 业务人员不看源码也能说明典型流程和异常；
- 培训人员能用流程图和一笔示例完成讲解；
- 页面字典能回答新增、编辑、导入、查询等实际问题；
- 技术取证不会干扰正文阅读，但在内部可追溯。
