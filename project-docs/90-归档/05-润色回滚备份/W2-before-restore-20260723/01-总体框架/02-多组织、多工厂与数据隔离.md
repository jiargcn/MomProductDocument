# 多组织、多工厂与数据隔离

> 状态：组织层级仍为**参考设计**；当前表结构、权限过滤和隔离机制待核验，文中结论未核验前不可当实施定论。
> 阅读对象：实施架构、权限设计、跨工厂业务讨论；**不是**售前 15 分钟浅层主链（被问到组织隔离时再打开）。

本页要说清：同一套产品里，租户 / 组织 / 工厂如何切开主数据与业务数据，避免「能进菜单就等于能看全部数据」。

## 设计目的

在同一产品中支持多组织、多工厂、多仓库的业务运行，同时明确主数据、业务单据、库存与权限的隔离范围，避免仅凭菜单权限判断数据可见性。

## 参考设计（待核验）

旧版规划书建议采用「企业/组织—工厂（站点）—部门」的层级：工厂作为基础数据和业务数据的重要隔离维度，部门作为组织树及责任归属维度。该模型只作为核验起点，**不代表**当前数据库的真实层级。

## 必须确认的规则

| 主题 | 待确认内容 |
| --- | --- |
| 租户与组织 | 租户、企业、组织、工厂、站点、部门的实际对象和上下级关系 |
| 数据隔离 | 各主数据、单据、库存、日志按何种维度隔离或共享 |
| 跨工厂业务 | 调拨、委外、供应商协同、集中主数据等跨边界场景 |
| 权限联动 | 角色、岗位、仓库/库位及数据范围如何共同决定可见与可操作数据 |
