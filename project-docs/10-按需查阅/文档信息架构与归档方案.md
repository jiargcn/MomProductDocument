# 文档信息架构与归档方案

> 状态：2026-07-21 按方案 A 重整物理目录。  
> 目的：把「日常要盯的」和「过程归档」分开，降低 `project-docs` 认知负担。

## 1. 原则

1. **日常入口唯一**：只维护 `00-需关注/`。
2. **证据按需**：字段证据、菜单树等放 `10-按需查阅/`，不塞进日常入口。
3. **过程归档**：已完成阶段的清单、审计、映射表进 `90-归档/`，文首标明「归档文件」。
4. **正式站点隔离**：`project-docs` 不进 MkDocs；证实结论再写入 `docs/`。

## 2. 物理目录

```text
project-docs/
├─ README.md
├─ 00-需关注/          # 日常必看（≤6 个核心文件 + 本目录 README）
├─ 10-按需查阅/        # 执行方案、证据库、图示任务、本方案
└─ 90-归档/
   ├─ 01-历史总控过程/
   ├─ 02-阶段执行档案/
   ├─ 03-结构治理过程/
   └─ 04-历史方法闸门/
```

## 3. 需关注清单

见 [`../00-需关注/README.md`](../00-需关注/README.md)。

## 4. 归档规则

- 阶段已关闭、不再驱动「下一步」的过程稿 → `90-归档/` 对应子目录。
- 每个归档 Markdown 文首加：`>【归档文件】...`
- 归档默认只读；复盘时可引用，不可替代当前总控。
