## 4. 证据规则与注意事项

### 4.1 证据优先级

1. 测试环境实际行为（后续补充）。
2. `dev` 分支前后端源码。
3. DDL。
4. 菜单清单与前端路由。
5. 历史产品规划书：仅提取稳定设计理念或发现待核验点，不能单独证明当前实现。

### 4.2 必须遵守

- 只修改本仓库的 Markdown、`mkdocs.yml` 和必要链接配置；不修改前后端源码、DDL 或测试数据。
- 字段技术名必须由 DDL/DO/DTO/VO/前端字段共同核对；没有证据写“待核验”。
- 前端 `hasPermi`、菜单可见或按钮展示，只能证明前端配置，**不能**证明后端接口已强制授权。
- 业务状态、动作、导入、终端、第三方接口均逐业务取证；只复用章节结构，不复用具体结论。
- 标签/条码/打印当前可有 DBC/WMS 入口，但产品目标归属为 INFRA；必须同时写明“当前入口”和“目标归属”。
- `isRecycled` 的业务语义为“回收件”；前端显示“标准件”是已确认错误。
- 不确定事项默认登记差距总账后继续。仅在大规模重命名/移动、修改源码、使用敏感凭据、不可逆操作，或两种方案都会显著改变文档架构时再请求确认。

### 4.3 提交与推送

- 每个提交聚焦一组文档变更，提交信息使用 `docs: 中文摘要`。
- 提交前至少执行 `git diff --check` 与 `mkdocs build`；涉及 `.linkmap.json` 时先运行 `python scripts/link-interfaces.py` 并人工复查 diff。
- 当前远端推送尚需授权；不要因推送失败而回退、重置或破坏本地提交。
