# 两阶段文档建设路线图

> 决策日期：2026-07-16
> 范围说明：本文的 WS 按已确认口径指 WMS。

## 1. 目标

先在 DBC 与 WMS 两个最能代表“主数据维护 + 仓储业务流转”的模块中，验证文档是否真正可读、可讲、可查、可继续扩写；标准未经验证前，不把同一问题复制到其它模块。

## 2. 第一阶段：DBC 与 WMS 的宏观说明和大纲验证

### 2.1 范围

**只允许处理：**

- `docs/04-DBC-主数据管理/`；
- `docs/05-WMS-库房管理/`；
- 为解释这两个模块所必需的共享业务模型页面；
- 与本阶段有关的 `project-docs/` 模板、任务和证据底稿。

**明确不处理：**System、Infrastructure、MES、QMS、EAM、ANDON、SCP、PS、API 及其已有页面。它们已经建立的首轮占位保留，但在第一阶段不新增、不回填、不重构。

### 2.2 交付内容

| 层级 | 需要形成的成果 | 完成标准 |
| --- | --- | --- |
| 共享认知 | DBC/WMS 相关的申请、任务、记录、库存挂接等宏观说明。 | 能解释对象为什么存在、如何协作，不从代码实现开始讲。 |
| 模块说明 | DBC 与 WMS 各一份可用于入门培训的模块概述。 | 包含模块目的、角色、前置数据、典型流程、跨模块关系、常见异常、学习顺序和术语入口。 |
| 业务分组 | 每个业务分组都有完整业务大纲。 | 标题一次性齐全；未填内容使用 `【待补充】`、`【图示占位】`、`【截图/示例数据占位】`。 |
| 样板页 | 至少重构物料基本信息、采购收货两个样板。 | 先讲业务与操作，再提供按需页面字典；技术取证不干扰正文。 |
| 图示任务 | DBC/WMS 的关键流程、状态、对象关系都有图示/截图/示例占位。 | 每个占位说明图或截图要表达什么、后续需要什么资料。 |
| 页面分层 | 为复杂页面明确主文档与详细参考的分工。 | 主文档讲业务主线，详细参考讲具体维护/执行/查询；不机械拆页。 |

### 2.3 工作顺序

1. [x] 用物料基本信息、采购收货及其详细参考完成 P0 双样板复审，固化正式页面标准。
2. 改写 DBC、WMS 模块首页：先建立业务入门地图，再补模块全景图和学习顺序。
3. 为 DBC 八个、WMS 十三个业务分组建立完整章节大纲和多媒体占位。
4. 为每个复杂叶页判断采用“一页完成”还是“主文档 + 详细参考”，但不在本阶段填满字段和技术细节。
5. 用新人培训、业务讲解、快速查询三个视角抽查模块首页、分组页和样板页。
6. 根据审阅结果修正模板、目录和大纲，再决定是否准入第二阶段。

具体分派顺序、分组范围和完成检查见[第一阶段大纲建设执行清单](第一阶段大纲建设执行清单.md)。

### 2.4 第一阶段禁止事项

- 不继续做 System/Infrastructure 的 RBAC、平台能力或其它模块页面回填。
- 不把 DDL、DO、DTO、VO、接口调用链作为正文主体。
- 不因为“先占位”只留下空标题、空表或“待核验”字样。
- 不在未通过样板审阅前，继续批量扩写 DBC/WMS 的全量字段与技术细节。

## 3. 第一阶段准入评审

以下全部满足，才进入第二阶段：

- [ ] DBC 与 WMS 模块首页能各自完成一次 10 分钟入门讲解。
- [ ] 业务人员能根据两个样板页讲清业务目的、角色、典型流程、状态与异常。
- [ ] 每个 DBC/WMS 业务分组页面都有完整大纲和明确站位，而不是技术取证表。
- [ ] 主文档与详细参考的边界在抽查页面中保持清晰，不出现大段重复或技术底稿外溢。
- [ ] “我要学习”和“我要查询”两类读者都能找到入口。
- [ ] 图示、截图、示例数据的后续任务可直接分派。
- [ ] 技术证据与产品差距已从正文叙事中隔离，不影响正常阅读。
- [ ] 产品负责人确认文风、层次和目录结构符合预期。

## 4. 第二阶段：按已验证标准大范围扩写

第二阶段不是机械复制。先把第一阶段通过的模板、样板和审阅结论固化，再按优先级推进：

1. 补齐 DBC/WMS 的页面字典、规则、截图和示例；
2. 推进 System/Infrastructure 的业务化概述与公共能力说明；
3. 依次推进 MES、QMS、EAM、ANDON、SCP、PS、API；
4. 每一模块仍先做模块说明与完整大纲，后做详细规则和技术核验。

任何新模块若再次出现“正文像技术底稿”的问题，应停止该模块扩写，回到第一阶段的样板和模板修正，而不是继续扩大范围。
