> 状态：第一阶段执行中。P0 双样板和正式页面模板已定稿；现只对 DBC/WMS 做模块说明、完整大纲和可分派站位，其他模块的逐页取证暂缓。
> 前置：BATCH-01 至 BATCH-04 首轮占位已完成。
> 目的：从“结构占位”转入“证据取证 + 业务规则回填”，避免文档长期停留在待核验状态。

## 1. 阶段目标

本阶段先验证 DBC/WMS 的表达标准：先形成可培训的模块说明和完整业务大纲，再按需补充页面字典。技术取证继续作为内部依据，但不直接主导正式页面的章节和叙事。

| 目标 | 完成标准 |
| --- | --- |
| 字段真实性 | 字段技术名、中文名、类型、长度、枚举、来源与 DDL / DO / DTO / VO / 前端配置一致。 |
| 新增约束 | 新增必填、唯一性、默认值、选择器数据源和筛选条件有证据。 |
| 编辑限制 | 可编辑字段、不可编辑字段、状态限制、引用限制和删除限制有证据。 |
| 导入规则 | 导入模板字段、系统填充字段、校验规则和错误回执明确。 |
| 列表与详情 | 列表字段顺序、筛选条件、详情分组、快速跳转和关联 Tab 明确。 |
| 状态动作 | 按按钮、接口、状态前置、数据影响、权限和审计形成矩阵。 |
| 终端操作 | PDA、线边端、工位端入口、扫码规则、状态前置和数据影响明确。 |
| 接口挂接 | 第三方系统、触发业务、同步方向、幂等、重试、失败补偿和日志明确。 |
| 图示与示例 | 复杂业务具备流程图、状态图、截图、示例数据或示例流转。 |

## 5. 问题处理

默认采用“**登记问题、继续推进**”的工作方式：单页存在待确认项时，只要不影响其它页面的事实取证和结构建设，就登记到《问题与待确认总账》，在页面保留简短的待确认标识，并继续处理下一项。问题记录应包含证据基线、影响范围、所需补充资料和建议验证动作，避免后续重复追问。

| 问题类型 | 处理方式 |
| --- | --- |
| 字段缺证据 | 页面标记“待核验”，登记到字段技术名真实性校正任务。 |
| 规则冲突 | 页面保留证据来源，登记到问题与待确认总账。 |
| 实现不完整 | 不写成缺陷结论，记录当前实现、影响和后续确认人。 |
| 需要大范围改名/移动 | 登记为结构治理任务，不在取证阶段顺手处理；继续处理不依赖该改名的页面。 |
| 需要源码修改或生产数据 | 登记为实现或测试验证任务，不属于当前文档仓库工作范围；继续完成源码可证实的部分。 |
| 业务口径无法安全推断 | 仅该口径标记“等待确认”；先完成字段、入口、页面行为等独立内容。 |

只有以下情况才需要向产品负责人发起确认：同一概念存在互斥且会改变培训结论的业务定义；继续写下去会把猜测误写为当前实现；或下一批所有可执行任务均依赖同一项外部资料。其余问题均不构成本轮中断条件。
