## 3. 必须遵守的约定

1. 正式 `docs/` 不写 DDL、DO、DTO、VO、Controller、Service、表名或源码路径。
2. 每个业务页面都必须覆盖维护与查询；仅在符合统一判定表的情况下增加“主文档名称-维护与查询参考”。
3. 未证实的状态、权限、导入、终端或跨模块规则，登记到唯一总账后继续推进；只有会导致大范围返工、不可逆变更或授权风险的问题才暂停确认。
4. 标签、条码和打印按“当前业务入口 + 目标基础设施归属”分别说明，不另建孤立模块。
5. 每次提交前运行 `git diff --check` 和 `mkdocs build`；既有跨模块链接告警不属于本轮失败，新增告警必须处理或登记。
6. 第二阶段仍先模块概述与完整大纲，后做详细规则；若再现“正文像技术底稿”，停止该模块扩写并回改模板。
