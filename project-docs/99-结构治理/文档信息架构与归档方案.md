# 文档信息架构与归档方案

> 状态：内部工作资料已从产品站点物理迁出。
> 目的：把面向读者的产品文档与建设过程材料分开，既保证正式内容可用于培训和业务查询，也保留可追溯的协作底稿。

## 1. 当前问题与整理原则

此前 `docs/15-版本路线图` 同时存放总计划、模板、问题清单、资料模板和已完成批次的执行记录。它们的更新频率、阅读对象和保留目的不同：既会把内部未确认内容直接呈现给站点读者，也会使执行者难以判断“下一步做什么”。

采用以下五类文档：

1. **工作总控**：持续决策与当前任务。
2. **标准与方法**：可复用的页面结构和取证规则。
3. **问题与待确认**：产品差距、证据冲突、资料缺失和待验证项。
4. **资料与证据**：填写模板、字段底表等可追溯输入。
5. **阶段执行档案**：已完成阶段的过程证据，默认只读。

## 3. 已实施的整理

- 内部工作资料按“工作总控、标准与方法、问题与待确认、资料与证据、阶段执行档案”分类；
- 25 个内部 Markdown 文件已从 `docs/15-版本路线图/` 迁至仓库根目录 `project-docs/`，并保留分类结构；
- MkDocs 的“版本路线图”只保留面向读者的正式入口，不再导航内部任务、证据或问题底稿；
- 产品页面原先指向内部模板或问题总账的链接，已改为不带链接的内部引用，避免站点读者将未审定内容误认为正式结论；
- 迁移前后均应重算相对链接，并执行 Markdown 链接检查、`git diff --check` 和 `mkdocs build`。

## 4. 当前物理目录结构

下列结构已落地。后续新增文件应先判断其读者与稳定性，避免将建设过程材料重新放入产品站点。

```text
docs/
└─ 15-版本路线图/
   └─ index.md                 # 正式版本能力、限制、变更与路线图

project-docs/
├─ README.md                   # 内部工作台入口
├─ 01-工作总控/
├─ 02-标准与方法/
├─ 03-问题与待确认/
├─ 04-资料与证据/
├─ 90-阶段执行档案/
│  ├─ P0-WMS纵向试点/
│  ├─ BATCH-01/
│  ├─ BATCH-02/
│  ├─ BATCH-03/
│  └─ BATCH-04/
└─ 99-结构治理/
```

### 本次迁移执行的强制步骤

1. 先生成“旧路径 → 新路径”映射表，确认无同名冲突。
2. 修改 `mkdocs.yml`、所有相对链接和产品页面中的内部引用；涉及跨模块链接时同步复查 `.linkmap.json`。
3. 运行 `python scripts/link-interfaces.py`（如映射受影响）、`git diff --check`、`mkdocs build`。
4. 检查全站链接、新增告警、导航顺序和历史档案可访问性。
5. 单独提交为“文档信息架构迁移”，不要与业务内容回填混在同一提交中。

## 5. 日常使用规则

- 开始工作：只看“工作总控”中的续作指南和逐页任务清单。
- 写页面：只看“标准与方法”中的模板，不从执行档案复制具体字段、状态或权限结论。
- 发现不确定项：优先登记“问题与待确认”，然后继续其它工作。
- 查历史原因：进入“阶段执行档案”；复盘结束后不把它替代当前总计划。
- 新增文档前先判断归属；若是一次性执行过程，默认进入阶段档案而非根目录。

## 6. 后续结构治理边界

本次迁移已影响 MkDocs 导航、跨页面相对链接和外部书签。后续如需再移动、合并或重命名这些目录，仍应先建立路径映射并单独完成链接、导航和构建校验；不要与业务内容回填混在同一提交中。

正式产品页面不得直接链接 `project-docs/`。当内部问题、底稿或任务得到证实并需要对使用者说明时，应提炼为稳定结论，再写入相应业务页面或 `docs/15-版本路线图/` 的正式版本说明。
