# 缺口标记写法规范

> 适用对象：文档建设与评审人员（快速扫未完成项）。  
> 正式读者也会看到标记，但语气按「建设中」理解，不是产品缺陷告警。  
> 默认**带 emoji 前缀**；中文关键词必须保留，便于全文搜索。

## 1. 四类标记（不要再拆）

| 类型 | 默认文案 | CSS / Admonition | 何时用 |
| --- | --- | --- | --- |
| 待核验 | `🔍 待核验` | `doc-gap--verify` / `!!! warning` | 已有说法，缺测试环境或源码复核 |
| 待确认 | `❓ 待确认` | `doc-gap--confirm` / `!!! warning` | 缺产品/开发口径，不能当培训结论 |
| 待补充 | `📝 待补充` | `doc-gap--fill` / `!!! tip` | 章节在，内容还空（含原「待添加」） |
| 媒体占位 | `📷 截图占位` / `📐 图示占位` | `doc-gap--media` / `!!! example` | 缺截图或图示；示例数据也用 example |

高风险少用：`⛔ 勿按此培训` → `doc-gap--danger`。

## 2. 句中 / 表格（徽章）

```markdown
服务层已按粒度合并，DDL 复合唯一约束 <span class="doc-gap doc-gap--verify">🔍 待核验</span>。

| 项 | 说明 |
| --- | --- |
| 导入必填 | 与保存模型不一致 <span class="doc-gap doc-gap--confirm">❓ 待确认</span> |
```

## 3. 整段 / 媒体（Admonition）

```markdown
!!! warning "🔍 待核验"
    ……尚未用测试环境复核。

!!! tip "📝 待补充"
    ……尚未按现场话术写完。

!!! example "📷 截图占位"
    新增页：标出……；使用脱敏测试数据。

!!! example "📐 图示占位"
    ……关系图（业务对象名）。
```

旧写法 `【截图占位：…】`、`【图示占位：…】`、`【待补充：…】`、`【示例数据/示例任务占位：…】` **不要再用**；`docs/` 存量已批量改为 admonition。

## 4. 禁止

- 不用 CSS 自动高亮正文里任意「待确认」字样（会误伤章节标题「限制与待确认」）。
- 不用纯红字铺全文；大红留给 `doc-gap--danger`。
- 徽章里不要省略中文关键词（只写 emoji 不可搜）。

## 5. 润色时必须保留（与方案 §7.1 禁删清单一致）

正式页中的四类缺口标记（含 emoji / admonition / 徽章）、示例数据占位、以及「待确认 / 待核验 / 待完善 / 待补充」等站位，**禁止**在「润色 / 可读性优化 / 去工程腔」时删掉或改没——它们用于快速定位待完善项。此前禁止的是「售前勿进」「本刀」「北极星」等过程黑话，**不是**禁止本规范中的待确认/截图占位。若与润色方案冲突，**以「必须保留」用户硬裁决为准**（见 [文档润色与可读性优化方案](文档润色与可读性优化方案.md) §7.1）。

## 6. 与问题总账

能编号的写 `GAP-xxx` + 短说明；需要醒目扫读时再加徽章/告示框，二者可并存。
