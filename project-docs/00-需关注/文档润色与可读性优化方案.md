## 0. 初衷与硬裁决（先读这段）

**初衷：** 正式业务文档要**更容易理解**，让目标读者**愿意打开、愿意用、愿意读完**。

**北极星读者（叶页唯一）：** 测试 + 实施 + 运维（并列）。运维含答疑 / 日常维护 / 排障，细节需求与测试、实施同级；**禁止**降为「只看平台与部署」。

**分层不动：** 主文档 / 维护与查询参考 / `project-docs` 证据。润色不推翻分层，不另起第三套模板。

**非目标：** 不为售前降叶页深度；不为对照代码把技术证据拉回正式页；不借润色重做字段取证；不以「有趣」为 KPI。

---

## 1. 受众与表述边界

| 角色 | 地位 | 含义 |
| --- | --- | --- |
| 测试 / 实施 / 运维 | 叶页唯一北极星 | 验收按他们；运维深读业务叶页 |
| 新开发 | 顺带 | 同向，不另设文风 KPI |
| 售前 / 销售 | 只做浅层入口 | 首页 + 总体框架 + 模块「解决什么问题」；分流写在本方案/交接，**正式页禁驱赶语** |
| 研发 | 质量约束 | 只写已确认事实；冲突进总账；证据层对照 |

**正式 `docs/` 禁止**工程黑话：润色、北极星、Wave/Lane、短地图、本刀、售前勿进等。  
**正式 `docs/` 必须保留**待确认 / 待核验 / 待完善 / 截图·图示·示例站位（见 §4 禁删清单）。过程黑话 ≠ 缺口站位。

## 2. 成功标准（顺序不可颠倒）

叶页润色先过 **A**，再过 **B**；只过 B 不过 A = **未达标**。

### A. 愿读（一票门槛）

1. **30 秒**：开篇能抓住「这页解决什么、读完能得到什么」。
2. **一条主线**：能跟完「一笔典型业务」（文字 ± 图），不被大表/义务节打断。
3. **少废话**：无明显同义重复；分流表用**目的**分行（理解主线 / 落地配置 / 排障 / 操作细节），不要每页挂「测试｜实施｜运维」任务书。
4. **净增受控**：相对改前，主文档净增宜 ≤ 约 20%；超了须先压缩/合并旧段再加新意。配置影响与验证点**优先嵌进现有节**，少开平行一级标题。
5. **完成感**：能写则写「给定配置 → 期望行为」；截图未齐不阻塞，但主线不能只剩占位。

### B. 能用（内容门槛）

1. **测试**能据此设计验证场景（配置差异、状态门禁、异常、跨模块）。
2. **实施**能说清「改哪个配置 → 行为如何变」；无独立配置则写所受通例/上游约束。
3. **运维**能据此答疑 / 维护 / 排障（配置·状态·上下游），不要求只靠平台页。
4. 关键字段表服务主线（约 8～12 行内），完整表在维护参考。

### C. 治理（非文风 KPI）

正式页只写已确认业务事实；未证实标 `❓` 或进总账；禁止脑补与技术证据腔。

### D. 入口 / 模块 / 分组 `index`（与叶页不同深，但同样要改正文）

入口页**不是**叶页定标的缩水版，也**不是**「只改阅读对象一行」。目标：让读者愿意从地图走进下一层。

**必须在正文改写或增强（至少满足 3 项，禁止只改元数据交差）：**

1. **开篇职责段**：用读者语言重写「解决什么问题 / 读完能判断什么」（改写旧句，不只加一句钩子）。
2. **目的导向入口**：有「如何使用 / 从哪开始读」表或等价段落——按目的分行（理解定位 / 维护配置 / 排障查询 / 进分组细读），不要角色任务书。
3. **短边界或配置依赖**：与上下游模块、主数据/执行边界；可嵌进现有「关键边界」节，3～8 行即可。
4. **学习或维护顺序**（若页上已有）：改写增强可读性，或补一句「不确定时先看哪」；**禁止**删掉既有顺序表。

**明确不算达标：**

- 只改「阅读对象」行、页眉基线、个别用词，正文结构与导读无实质变化。
- 以「结构已齐全 / 已有分组表」为由跳过正文改写。
- 用短地图或新骨架整页替换旧入口（仍一票否决）。

入口页净增宜 ≤ 约 15%；优先改写压缩，少开平行一级标题。叶页专属的「建议验证点 / 关键字段表」**不**作为入口页必做项。

## 3. 怎么改一页（手法）

### 3.1 信息义务，不是标题义务

下列信息**必须在页内讲清**，但**不必**各占一个一级标题；可嵌入「业务目的 / 一笔典型业务 / 关键判断 / 常见问题」等现有节：

| 必须讲清的信息 | 说明 |
| --- | --- |
| 开篇承诺 | 问题 + 读完能掌握什么（两句内） |
| 主文档 vs 维护参考分流 | 目的导向，短表即可 |
| 一笔典型业务 | 触发 → 处理 → 结果；含关键分支 |
| 使用前准备 / 关键判断或规则 | 按页已有结构增强 |
| 配置如何影响行为 | 有则写清；无则写所受约束 |
| 建议验证点 | 短列表，非完整用例集 |
| 影响与异常入口 | 便于运维；可链维护参考 |
| 关键字段业务角色 | 仅主线；禁止正文用 P1–P14 当读者标题，禁止「见上表 P6」这类内部代号 |

### 3.2 推荐主脊骨（业务叶页）

目的 → 分流 → 准备 → **一笔业务（含写实示例）** → 关键判断 → **穿插配置影响** → 短验证点 → 影响/异常 → 限制与站位。

配置页可略调顺序，但仍是「一条脊骨」，避免「典型业务 / 逻辑要点 / 配置作用 / 维护重点 / 影响 / 异常」六套平行大纲说同一件事。

### 3.3 密度

允许为讲清配置/门禁而略密；**不允许**用「密一点」合理化同义堆叠或无主线的义务节加法。完整字段表下沉维护参考。

### 3.4 入口 / 模块 / 分组 index 手法

```text
职责开篇（改写）→ 目的导向「从哪读」→ 保留/理顺学习或维护顺序与分组表
→ 短边界或配置依赖（嵌现有节）→ 站位与待确认原样保留
```

- **改写优先于追加**：旧「模块解决什么问题」「谁会使用」能改就改成更清楚的职责 + 目的分流，避免「旧表不动 + 再挂一张新表」。
- **「结构已齐」≠ 润色完成**：有分组表、mermaid、边界表只说明骨架在；仍须检查开篇是否愿读、分流是否目的导向。
- **禁止交差口径**：派工与自检不得把「仅更新阅读对象」写成完成。

## 4. 硬禁令（一票否决）

1. **润色 ≠ 重构：** 禁止短地图 / 新骨架**整页覆盖**旧正文。
2. **禁止掏空既有信息：** 学习/维护/操作顺序、业务目的、关键字段业务角色、FAQ、对象边界等**信息不得删没**；允许改写表达、调整顺序、**合并同义节**。
3. **增量 = 以旧版为底增强信息：** 允许改写、压缩、合并；**不是**「只能追加、不许变短」。变短且信息仍在 = 合格润色。
4. **禁删清单（站位与待确认）：** 截图/图示/示例数据站位、`待补充的图示与示例`、待确认/待核验/待完善及 [缺口标记写法规范](缺口标记写法规范.md) 醒目标记（含 `📷`/`📝`/`❓` 等）——禁止借润色删掉或改没。
5. **发现整页覆盖：** 立即停写 → 恢复旧底 → 再按本方案改；未恢复前不开下一内容波。

## 5. 单页验收清单

**A 愿读**

- [ ] 30 秒抓住用途与读完收获
- [ ] 能跟完一笔主线；配置/验证未另起平行大纲抢戏
- [ ] 无明显同义重复；分流为目的导向
- [ ] 净增受控（超 20% 已先压缩）；未把「加节齐全」当成通过
- [ ] 站位与待确认类标记仍在

**B 能用**

- [ ] 配置影响（或所受约束）说得清
- [ ] 有短验证点
- [ ] 测试/实施/运维抽查能干活（开场景 / 讲配置 / 答疑排障）
- [ ] 关键字段未膨胀成字典；无技术证据腔；未为售前砍深度；未把运维限缩为仅平台

**否决：** 短地图整页覆盖；掏空既有信息；误删禁删清单项；**入口/模块 index 仅改阅读对象或元数据、正文无实质改写**。

**入口 / 模块 index 另检（方案 §2.D）：**

- [ ] 开篇职责段有实质改写（读完能判断什么）
- [ ] 有目的导向「从哪读 / 如何使用」（非角色任务书）
- [ ] 有短边界或配置依赖（嵌现有节亦可）
- [ ] 未以「结构已齐」跳过正文；学习/分组/站位仍在

## 6. 反模式（出现则退回）

1. 开篇字段全家福或「行为模式清单」。
2. 用空话代替「选错/配错会导致什么」。
3. 技术名 / 源码 / API / 权限注解进正式页。
4. 全文变成缺口堆砌（标记须留，但不要抢主线）。
5. 主文档与维护参考重复且无分流。
6. 售前路径链进维护参考或字段长表；或为售前砍配置/验证。
7. 只追加义务节、不压缩不改写 → 页更长更劝退。
8. 把「有趣/故事感」或「章节清单勾满」当成愿读通过。
9. 无依据脑补配置影响。
11. **只改阅读对象 / 元数据交差（入口页）：** 模块或分组 `index` 仅改页眉阅读对象、基线日期等，开篇职责与「从哪读」无实质改写 → 退回（见 §2.D）。
12. **以「结构已齐」跳过入口润色：** 有分组表/顺序表不等于愿读达标。

## 7. 执行方式（摘要）

- 方式：样板定标 → 分波推进；同波路径互斥；波间串行验收。
- 路径表与勾选状态：**只写在** [文档润色-波次计划](文档润色-波次计划.md) 与 [交接](当前工作交接与续作.md)，避免本方案正文被过程状态撑胀。
- 子代理交付：改动列表 + §5 自检（先 A 后 B）+ 未决 `❓` + 确认未触犯 §4。
- 波末：人读抽查愿读（不只看 diff）+ `git diff --check` + `mkdocs build`。

### 派工提示词骨架

```text
你是文档润色子代理。只处理指定独占路径。
初衷：让测试/实施/运维更容易理解、愿意使用文档（见 文档润色与可读性优化方案.md）。
成功标准：先过「愿读」再过「能用」；禁止只加节勾清单。
若改入口/模块/分组 index：必须按方案 §2.D / §3.4 做正文实质改写（开篇职责 + 目的导向从哪读 + 短边界）；「只改阅读对象」= 未达标；「结构已齐」不能作为跳过理由；仍禁短地图整页覆盖。
若改叶页：以旧版为底；允许改写/压缩/合并同义节；配置影响与验证点优先嵌入现有节；净增宜≤约20%。
硬禁令：禁止短地图整页覆盖；禁止掏空既有信息；禁止删除截图/示例/待确认等站位。
正式页禁止工程黑话与售前驱赶语；禁止正文写「见上表 P?」。
禁止修改：其它路径、mkdocs.yml、.linkmap.json、问题总账。
交付：改动列表 + 愿读/能用自检（入口页另勾 §2.D）+ 未决❓ + 确认未触犯硬禁令。
不要自行开下一波。
```
