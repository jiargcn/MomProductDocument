**北极星（唯一）：** 把正式业务页写成给 **测试 + 实施 + 运维**（三者并列）用的「业务行为说明书」——功能范围清楚、逻辑可讲、配置可落地、验证可开单；运维同样能据此**答疑、日常维护与排障**，对系统细节的了解需求与测试、实施同级。

**Goal（服从北极星）：** 主文档可扫读、愿意读完；维护参考可深查；避免字典感与研发底稿腔。不以「有趣/娱乐化」为 KPI。

**Architecture:** 不推翻现有三层（主文档 / 维护与查询参考 / `project-docs` 证据）。在既有结构上叠加：读者路径分流、文风硬约定、配置与验证点义务、分波润色与验收门禁。

**Non-goals（防范围膨胀）:**

- 不借润色重做字段取证，不恢复正式页技术字段清单。
- 不重写 `project-docs` 证据库文风。
- 不把全部截图作为润色前置（无图时用写实示例顶上）。
- 不另起第三套页面模板；约定与检查清单并入现有模板。
- 不把正式 `docs/` 写成 API/源码说明书。
- **不为售前降低叶页深度**；不为「研发对照代码」把技术证据拉回正式页。
- **不把运维降级为「只看平台与集成」或仅部署/监控**；运维与测试、实施同属 P1 深读核心。
- 不追求「售前/研发对照与 P1 在同一页上都被优化到最优」。

## 0. 受众冲突裁决（必须遵守）

多类人都会用到文档，但**不能**把方向不同的读者同时当作叶页润色的优化对象，否则方向会摇摆。**测试、实施、运维三者并列为核心目标受众（P1）**——对系统细节了解需求同等高；售前与研发对照仍不得把叶页带偏。

| 角色 | 在本方案中的地位 | 含义 |
| --- | --- | --- |
| **测试 / 实施 / 运维（P1）** | **唯一润色北极星（三者并列）** | 叶页验收、文风密度、章节义务全部按他们定；运维含答疑、日常维护、排障，**不仅是部署监控** |
| **新开发（P2）** | **顺带受益** | 与 P1 同向（边界/逻辑/主对象）；不另设文风；不单独加 KPI |
| **售前 / 销售（P3）** | **只做浅层入口** | 仅首页 + 总体框架 + 模块「解决什么问题」；**叶页润色不为售前降维** |
| **研发自身（P4）** | **质量约束，非文风 KPI** | 正式页只写已确认业务事实；冲突进总账；技术对照在证据层。不把「反向观察源码」写成润色成功标准 |

**可读性定义（替换「有兴趣」）：** 结构清楚、可扫读、少废话、有主线 → 读者**愿意读完**。不追求故事腔或趣味化。

**主文档密度裁决：** 允许为配置影响、状态门禁、跨模块影响与验证点而「干一点、密一点」；完整字段表仍下沉维护参考。对 P1，「清楚、能用」优先于「轻薄好看」。

### 0.1 工程用语边界（`project-docs` vs 正式 `docs/`）

下列用语**仅用于** `project-docs/` 工程文件、交接、派工与子代理提示词，**禁止**写入正式 `docs/` 业务正文：

| 工程用语 | 用途边界 |
| --- | --- |
| 润色、北极星 | 内部方案与验收口径 |
| Wave / W0–W7、Lane | 波次派工与闸门 |
| 短地图 | 失败教训与结构禁令讨论 |
| 读者代号 P1–P4 | 方案内受众裁决（勿与字段行为模式 P1–P14 混淆进正文标题） |

正式 `docs/` **只用读者语言**：测试人员、实施人员、运维人员、主文档、维护与查询参考、业务分组、学习/维护/操作顺序等。验收抽查时用「测试 / 实施 / 运维视角」表述即可，不必把工程黑话带进页面。

**勿混淆：** 上文禁止的是「售前勿进」「本刀」「北极星」「Wave/Lane」等**过程黑话**；**不是**禁止正式页中的「待确认 / 待核验 / 待完善 / 截图占位」等缺口站位（见 §7.1 禁删清单与 [缺口标记写法规范](缺口标记写法规范.md)）。

### 0.2 售前表述边界

- 正式 `docs/` **禁止**「售前勿进」「售前请停在…」等对读者的驱赶语或禁入告示。
- 浅层入口用中性表述，例如「了解产品全景」「模块解决什么问题」「从哪开始读」。
- **售前分流路径**只写在本方案与内部交接/派工中沟通；不写成面向站点读者的禁区说明。
- 「勿默认把售前路径链到维护与查询参考」属于**写作 / 验收规则**（见 §5.5、§6、§7），由作者与验收执行——**不要**把它改写成页面对读者的「禁止进入」告示。

## 1. 谁会用到文档（说明用，非并行 KPI）

| 代号 | 读者 | 核心诉求 | 主要阅读路径 | 润色时怎么对待 |
| --- | --- | --- | --- | --- |
| P1 | **测试、实施、运维（并列）** | 有哪些功能；逻辑如何走；**配置/状态如何改变行为**；跨模块影响。运维另含：**日常问题答疑、系统维护、排障**（配错/状态卡死/上下游连带） | 业务主文档 + 维护参考 + 策略/单据/开关及通例；基础设施/系统管理作**补充**（部署监控），**不是**运维的唯一入口 | **按此写叶页** |
| P2 | 新加入开发 | 快速吃透模块业务细节与边界 | 模块首页 → 分组 → 主文档 → 共享模型；争议时查证据 | 不另定标准；吃同一套主文档 |
| P3 | 售前、销售 | 相对笼统，便于对外介绍 | 「认识系统」+ 模块「解决什么问题」 | **只做阶段 1 浅层路径**；勿链维护参考 |
| P4 | 研发团队 | 与实现一致；发现文档/代码差距 | 正式结论 + [问题总账](问题与待确认总账.md) + 按需证据 | 变更纪律与缺口治理；**不优化正式页文风给对照代码用** |

现场业务用户、培训师可以是写清楚之后的受益者，但不是本方案排序依据。

## 2. 成功标准（可抽查）

**叶页润色（主标准，只看 P1——测试 / 实施 / 运维同级）：**

1. **测试**：能据此设计验证场景（含配置差异、状态门禁、异常与跨模块影响）。
2. **实施**：能说清「改哪个配置 → 流程/可选范围/现场行为发生什么变化」；无独立配置则写明受哪些通例/上游约束。
3. **运维**：能据此**回答现场日常问题**、做**系统维护**与**排障**——说清当前配置/状态意味着什么、卡在哪、改/查什么会影响上下游；**不要求**仅靠基础设施页完成业务答疑。
4. **可扫读**：有「一笔典型业务」主线；开篇两句说清本页干什么；关键字段表不吞噬主线。

**顺带（不单独为叶页加任务）：**

5. **新开发**：读模块首页 + 主文档后，能用业务语言说出边界、主对象、主状态与上下游协作点。

**浅层入口专用（阶段 1，不要求叶页）：**

6. **售前**：只走浅层路径即可做约 5～10 分钟介绍。

**治理约束（非文风验收项）：**

7. **一致性纪律**：正式页结论须为已确认业务事实；未证实标 `❓` 或进总账；禁止脑补；技术细节不进正式页。

## 3. 问题诊断（为何要润色）

| 问题 | 表现 | 对 P1 的伤害 |
| --- | --- | --- |
| 受众表述漂移 | 首页/模板曾把运维降为「只看平台」或仅部署，或摇摆于售前/培训 | 测试/实施/运维不知该读多深；运维答疑缺业务细节 |
| 字典感 | 字段/模式表过早过密；P 编号对读者无感 | 抓不住逻辑与配置主线 |
| 配置作用不够显式 | 业务类型、单据设置、开关、规则散落 | 配不准；难做配置差异验证；运维难排障 |
| 完成感不足 | 截图/示例占位多、主线弱 | 像未写完，不愿读完 |
| 缺口抢戏 | `❓`/GAP 打断叙事 | 分不清能依赖的事实与待确认 |
| 多目标并行 | 想同时讨好售前/研发对照/趣味 | 叶页深度与表达左右横跳 |

## 4. 必须纳入的维度（服务北极星）

| 维度 | 要解决的问题 | 润色时怎么做 |
| --- | --- | --- |
| 北极星锁定 | 多目标摇摆；运维被旁路 | 叶页只服务 P1（测试+实施+运维）；P3/P4 不进叶页验收；运维不降级为仅平台页 |
| 文风与扫读 | 底稿腔或空泛叙事 | 场景主线 + 逻辑要点；禁止开篇即大表；不追求趣味 |
| 术语治理 | 同义多写、术语裸奔 | 首次场景引入；共享锚点；正文不用 P1–P14 当标题 |
| 配置可实施性 | 配完不知道影响 | 主文档固定「配置如何起作用」 |
| 可测性 | 测试难落场景 | 主文档固定短「建议验证点」 |
| 信息分层 | 主文档/参考/证据混读 | 「如何使用本组文档」；完整表在维护参考 |
| 完成感 | 只有占位 | 流程 mermaid + 写实示例（给定配置→期望行为）；截图另轨 |
| 可找到性 | 跨模块不知下一步 | 上下游链接、学习顺序 |
| 可信度 | 事实与缺口混杂 | 正文短标；详情进总账；产品观察放文末 |
| 一致性 | 模块间骨架不齐 | 统一章节义务 + 单页验收清单 |
| 治理（P4） | 文档与实现脱节 | 只写已确认事实；冲突进总账；证据不进正式页 |
| 浅层入口（P3） | 售前误入深页 | 首页专用路径；叶页不为售前改写 |
| 优先级 | 全站齐改会烂尾 | 样板 → 高频主链 → 扩展模块 → 平台页 |
| 反模式门禁 | 改着又技术化或多目标化 | 清单一票否决 |

## 5. 文风、术语与章节硬约定

### 5.1 主文档必须有

1. **开篇承诺**（两句内）：本页解决什么问题；读完能掌握什么（功能范围 / 逻辑 / 配置影响）。
2. **如何使用本组文档**：分流主文档 vs 维护参考（给测试/实施/运维）。
3. **一笔典型业务**（文字 ± mermaid）：触发 → 处理 → 结果；含关键分支。
4. **使用前准备**、**业务逻辑要点**（状态/对象/跨模块边界）、**关键判断或规则**。
5. **配置如何起作用**：业务类型、单据设置、开关、规则、策略等；若无独立配置，写明「受某某通例/上游配置约束」。
6. **建议验证点**（短列表；不是完整测试用例集）。
7. **做完影响什么**、**异常与查询入口**（可链维护参考；便于运维答疑与排障）。
8. **关键字段业务角色**（仅主线；建议约 8～12 行内，超出下沉维护参考）。允许为讲清配置/门禁而略密，但不得变成字段全家福。

### 5.2 主文档禁止 / 限制

- 禁止 DDL / DO / DTO / VO / 源码路径 / 权限注解原文 / 未证实技术名。
- 禁止以 P1–P14 编号作为读者可见标题。
- 禁止开篇即大表；先主线与逻辑，后表。
- 禁止把问题总账全文铺进正文。
- 禁止为讨好售前而删掉配置/验证要点。
- 禁止为「方便对照代码」而写入技术证据。
- 禁止把运维读者指引成「只需看基础设施/部署页」而跳过业务叶页深读。

### 5.3 术语处理

- 先场景行为，再给出系统叫法；必要时链 `docs/02-业务模型/`。
- 中文业务名为主；英文/CODE 作辅。
- 后果解释优先于抽象定义。
- 模块首页可有短术语表；正文中途不插词典段。

### 5.4 维护参考与证据层

- **维护参考**：允许更高表密度；仍用业务语言，不写技术证据；运维日常维护与查询优先用此层。
- **`project-docs` 证据**：联调争议与实现对照时用；**不是**售前路径，也**不是**叶页润色对象。
- **与源码迭代一致（P4 约束）**：业务行为变更时先更新正式页结论或登记总账；禁止未证实推测补齐正式页。

### 5.5 阅读深度闸门

| 读者 | 应读到 | 不应默认读到 |
| --- | --- | --- |
| 测试 / 实施 / 运维 | 主文档全文 + 相关维护参考 + 配置通例（运维答疑/维护/排障同此深读）；部署监控可另补平台页 | 证据底稿（除非联调争议）；**勿把运维限缩为仅平台页** |
| 新开发 | 模块地图 + 主文档 + 共享模型 | 全站维护参考扫读 |
| 售前 / 销售 | 总体框架 + 模块「解决什么问题」 | 叶页细节、维护参考、字段长表、GAP |
| 研发（对照实现） | 正式结论 + 总账 + 按需证据 | 要求正式页长得像源码注释 |

## 6. 单页润色验收清单（叶页；执行时并入模板）

须同时满足（**抽查问测试 / 实施 / 运维视角，三者同级**）：

- [ ] 开篇承诺清晰（问题 + 读完能掌握什么）
- [ ] 有「如何使用本组文档」分流
- [ ] 有一笔典型业务主线（文字或图）
- [ ] 有「配置如何起作用」（或明确无独立配置及所受约束）
- [ ] 有短「建议验证点」
- [ ] 有准备 / 逻辑或判断 / 影响 / 异常中的可执行说明
- [ ] 关键字段表服务主线，未膨胀成字典页
- [ ] 术语首次有场景义或共享链接
- [ ] `❓` / GAP 不打断主线（短标或文末）
- [ ] 无技术证据腔
- [ ] 上下游至少各有一个有用链接（若存在协作）
- [ ] **抽查**：测试能据此列场景；实施能讲配置影响；运维能据此答疑/维护/排障（配置·状态·跨模块影响）  
- [ ] **否决项**：未为售前删深度；未为对照代码加技术细节；未把运维降为「只看平台」
- [ ] **否决项（结构 / §7.1）：** 未用短地图或新骨架整页覆盖；未删除既有章节/站位；确认为以旧版为底的增量合并
- [ ] **否决项（站位 / 待确认）：** 未借「润色/去工程腔/去 emoji」删掉或改没截图/示例站位、待确认/待核验/待完善及同类醒目标记（见 §7.1 禁删清单）

## 7. 反模式（出现则退回）

1. 开篇先贴字段全家福或「行为模式清单」。
2. 用「该字段参与上游关联」代替「选错/配错会导致什么」。
3. 把 DDL、类名、API path、权限注解写进正式页。
4. 正文变成 GAP/待确认清单（**≠** 删掉站位或待确认标记；标记须保留，只是不要让全文变成缺口堆砌）。
5. 主文档与维护参考重复且无分流。
6. 售前路径链进维护参考或字段长表。
7. 叶页为「售前也好懂」而砍掉配置/验证要点。
8. 为润色清空已证实业务结论（只许改写表达）。
9. 无依据脑补配置影响或过滤条件。

## 8. 推进方式：波次 + 子代理并行

### 8.1 方式

采用 **B：标准先行 + 样板定标 + 分波润色**，执行形态改为 **Wave（波次）**：

- 每一波固定 **3～4 条并行泳道（Lane）**，每条泳道交给 **1 个子代理**。
- **同波泳道之间路径互斥**，禁止改同一文件。
- **波与波串行**：上一波你验收通过后，才开下一波。
- 「完成感」（mermaid/写实示例）与「`❓`/GAP 减负」**并入各内容波的页内义务**，不再单开大波，减少回头改同一批文件。

### 8.2 并行硬规则（子代理必须遵守）

1. **只改本泳道「独占路径」**；可读其它页以便加链接，但不得改写非独占文件。
2. **不得改** `mkdocs.yml`、`.linkmap.json`、`scripts/`（除非该波明确授权的协调泳道）。
3. **不得改** `project-docs/00-需关注/问题与待确认总账.md`（缺口只在正文短标既有编号；新问题记入本泳道回报，由你或协调代理统一登记）。
4. 叶页一律按 **§5 / §6 / §7（含 §7.1）** 与北极星（测试/实施/运维并列）执行；截图未齐不阻塞；**入口/模块 `index` 同样禁止短地图整页覆盖**。
5. 泳道结束须提交：**改动文件列表 + 自检 §6 勾选结果 + 未决 `❓` 列表**；若动过结构，须声明「以旧版为底的增量合并」且未删 §7.1 所列既有章节。
6. **波末由你（或主会话）统一**：抽查 §6、§7.1、`git diff --check`、`mkdocs build`；通过后勾选本波闸门。
