# 文档润色与可读性优化方案

> **状态：待你确认后执行。** 本稿仅落盘方案与任务清单；未改 `docs/` 业务正文、未启动润色波次。
>
> 起草日期：2026-07-22  
> 修订：2026-07-22（按受众冲突裁决收束北极星）  
> 推进方式：**B — 标准先行 + 样板定标 + 分波润色**  
> 关联：[实体说明页标准模板](实体说明页标准模板.md)、[字段业务语义建设计划（已归档）](../90-归档/01-历史总控过程/字段业务语义建设计划.md)、[文档表达纠偏与重构方案](../90-归档/01-历史总控过程/文档表达纠偏与重构方案.md)、[待截图执行清单-2026-07-22](../10-按需查阅/标准与方法/待截图执行清单-2026-07-22.md)

**北极星（唯一）：** 把正式业务页写成给 **测试与实施** 用的「业务行为说明书」——功能范围清楚、逻辑可讲、配置可落地、验证可开单。

**Goal（服从北极星）：** 主文档可扫读、愿意读完；维护参考可深查；避免字典感与研发底稿腔。不以「有趣/娱乐化」为 KPI。

**Architecture:** 不推翻现有三层（主文档 / 维护与查询参考 / `project-docs` 证据）。在既有结构上叠加：读者路径分流、文风硬约定、配置与验证点义务、分波润色与验收门禁。

**Non-goals（防范围膨胀）:**

- 不借润色重做字段取证，不恢复正式页技术字段清单。
- 不重写 `project-docs` 证据库文风。
- 不把全部截图作为润色前置（无图时用写实示例顶上）。
- 不另起第三套页面模板；约定与检查清单并入现有模板。
- 不把正式 `docs/` 写成 API/源码说明书。
- **不为售前降低叶页深度**；不为「研发对照代码」把技术证据拉回正式页。
- 不追求「四类读者在同一页上都被优化到最优」。

## 0. 受众冲突裁决（必须遵守）

四类人都会用到文档，但**不能**同时作为叶页润色的优化对象，否则方向会摇摆。

| 角色 | 在本方案中的地位 | 含义 |
| --- | --- | --- |
| **测试 / 实施（P1）** | **唯一润色北极星** | 叶页验收、文风密度、章节义务全部按他们定 |
| **新开发（P2）** | **顺带受益** | 与 P1 同向（边界/逻辑/主对象）；不另设文风；不单独加 KPI |
| **售前 / 销售（P3）** | **只做浅层入口** | 仅首页 + 总体框架 + 模块「解决什么问题」；**叶页润色不为售前降维** |
| **研发自身（P4）** | **质量约束，非文风 KPI** | 正式页只写已确认业务事实；冲突进总账；技术对照在证据层。不把「反向观察源码」写成润色成功标准 |

**可读性定义（替换「有兴趣」）：** 结构清楚、可扫读、少废话、有主线 → 读者**愿意读完**。不追求故事腔或趣味化。

**主文档密度裁决：** 允许为配置影响与验证点而「干一点、密一点」；完整字段表仍下沉维护参考。对 P1，「清楚、能用」优先于「轻薄好看」。

## 1. 谁会用到文档（说明用，非并行 KPI）

| 代号 | 读者 | 核心诉求 | 主要阅读路径 | 润色时怎么对待 |
| --- | --- | --- | --- | --- |
| P1 | 测试、实施 | 有哪些功能；逻辑如何走；**配置如何改变行为** | 主文档 + 维护参考 + 策略/单据/开关及通例 | **按此写叶页** |
| P2 | 新加入开发 | 快速吃透模块业务细节与边界 | 模块首页 → 分组 → 主文档 → 共享模型；争议时查证据 | 不另定标准；吃同一套主文档 |
| P3 | 售前、销售 | 相对笼统，便于对外介绍 | 「认识系统」+ 模块「解决什么问题」 | **只做阶段 1 浅层路径**；勿链维护参考 |
| P4 | 研发团队 | 与实现一致；发现文档/代码差距 | 正式结论 + [问题总账](问题与待确认总账.md) + 按需证据 | 变更纪律与缺口治理；**不优化正式页文风给对照代码用** |

现场业务用户、培训师可以是写清楚之后的受益者，但不是本方案排序依据。

## 2. 成功标准（可抽查）

**叶页润色（主标准，只看 P1）：**

1. **测试**：能据此设计验证场景（含配置差异、状态门禁、异常与跨模块影响）。
2. **实施**：能说清「改哪个配置 → 流程/可选范围/现场行为发生什么变化」；无独立配置则写明受哪些通例/上游约束。
3. **可扫读**：有「一笔典型业务」主线；开篇两句说清本页干什么；关键字段表不吞噬主线。

**顺带（不单独为叶页加任务）：**

4. **新开发**：读模块首页 + 主文档后，能用业务语言说出边界、主对象、主状态与上下游协作点。

**浅层入口专用（阶段 1，不要求叶页）：**

5. **售前**：只走浅层路径即可做约 5～10 分钟介绍。

**治理约束（非文风验收项）：**

6. **一致性纪律**：正式页结论须为已确认业务事实；未证实标 `❓` 或进总账；禁止脑补；技术细节不进正式页。

## 3. 问题诊断（为何要润色）

| 问题 | 表现 | 对 P1 的伤害 |
| --- | --- | --- |
| 受众表述漂移 | 首页/模板曾摇摆于开发运维或现场培训 | 测试/实施不知该读多深 |
| 字典感 | 字段/模式表过早过密；P 编号对读者无感 | 抓不住逻辑与配置主线 |
| 配置作用不够显式 | 业务类型、单据设置、开关、规则散落 | 配不准；难做配置差异验证 |
| 完成感不足 | 截图/示例占位多、主线弱 | 像未写完，不愿读完 |
| 缺口抢戏 | `❓`/GAP 打断叙事 | 分不清能依赖的事实与待确认 |
| 多目标并行 | 想同时讨好售前/研发对照/趣味 | 叶页深度与表达左右横跳 |

## 5. 文风、术语与章节硬约定

### 5.1 主文档必须有

1. **开篇承诺**（两句内）：本页解决什么问题；读完能掌握什么（功能范围 / 逻辑 / 配置影响）。
2. **如何使用本组文档**：分流主文档 vs 维护参考（给测试/实施）。
3. **一笔典型业务**（文字 ± mermaid）：触发 → 处理 → 结果；含关键分支。
4. **使用前准备**、**业务逻辑要点**（状态/对象/跨模块边界）、**关键判断或规则**。
5. **配置如何起作用**：业务类型、单据设置、开关、规则、策略等；若无独立配置，写明「受某某通例/上游配置约束」。
6. **建议验证点**（短列表；不是完整测试用例集）。
7. **做完影响什么**、**异常与查询入口**（可链维护参考）。
8. **关键字段业务角色**（仅主线；建议约 8～12 行内，超出下沉维护参考）。允许为讲清配置/门禁而略密，但不得变成字段全家福。

### 5.2 主文档禁止 / 限制

- 禁止 DDL / DO / DTO / VO / 源码路径 / 权限注解原文 / 未证实技术名。
- 禁止以 P1–P14 编号作为读者可见标题。
- 禁止开篇即大表；先主线与逻辑，后表。
- 禁止把问题总账全文铺进正文。
- 禁止为讨好售前而删掉配置/验证要点。
- 禁止为「方便对照代码」而写入技术证据。

### 5.3 术语处理

- 先场景行为，再给出系统叫法；必要时链 `docs/02-业务模型/`。
- 中文业务名为主；英文/CODE 作辅。
- 后果解释优先于抽象定义。
- 模块首页可有短术语表；正文中途不插词典段。

### 5.4 维护参考与证据层

- **维护参考**：允许更高表密度；仍用业务语言，不写技术证据。
- **`project-docs` 证据**：联调争议与实现对照时用；**不是**售前路径，也**不是**叶页润色对象。
- **与源码迭代一致（P4 约束）**：业务行为变更时先更新正式页结论或登记总账；禁止未证实推测补齐正式页。

### 5.5 阅读深度闸门

| 读者 | 应读到 | 不应默认读到 |
| --- | --- | --- |
| 测试 / 实施 | 主文档全文 + 相关维护参考 + 配置通例 | 证据底稿（除非联调争议） |
| 新开发 | 模块地图 + 主文档 + 共享模型 | 全站维护参考扫读 |
| 售前 / 销售 | 总体框架 + 模块「解决什么问题」 | 叶页细节、维护参考、字段长表、GAP |
| 研发（对照实现） | 正式结论 + 总账 + 按需证据 | 要求正式页长得像源码注释 |

## 7. 反模式（出现则退回）

1. 开篇先贴字段全家福或「行为模式清单」。
2. 用「该字段参与上游关联」代替「选错/配错会导致什么」。
3. 把 DDL、类名、API path、权限注解写进正式页。
4. 正文变成 GAP/待确认清单。
5. 主文档与维护参考重复且无分流。
6. 售前路径链进维护参考或字段长表。
7. 叶页为「售前也好懂」而砍掉配置/验证要点。
8. 为润色清空已证实业务结论（只许改写表达）。
9. 无依据脑补配置影响或过滤条件。
10. 把「有趣/故事感」当成验收通过条件。

## 8. 推进方式：波次 + 子代理并行

### 8.1 方式

采用 **B：标准先行 + 样板定标 + 分波润色**，执行形态改为 **Wave（波次）**：

- 每一波固定 **3～4 条并行泳道（Lane）**，每条泳道交给 **1 个子代理**。
- **同波泳道之间路径互斥**，禁止改同一文件。
- **波与波串行**：上一波你验收通过后，才开下一波。
- 「完成感」（mermaid/写实示例）与「`❓`/GAP 减负」**并入各内容波的页内义务**，不再单开大波，减少回头改同一批文件。

### 8.2 并行硬规则（子代理必须遵守）

1. **只改本泳道「独占路径」**；可读其它页以便加链接，但不得改写非独占文件。
2. **不得改** `mkdocs.yml`、`.linkmap.json`、`scripts/`（除非该波明确授权的协调泳道）。
3. **不得改** `project-docs/00-需关注/问题与待确认总账.md`（缺口只在正文短标既有编号；新问题记入本泳道回报，由你或协调代理统一登记）。
4. 叶页一律按 **§5 / §6 / §7** 与北极星（测试/实施）执行；截图未齐不阻塞。
5. 泳道结束须提交：**改动文件列表 + 自检 §6 勾选结果 + 未决 `❓` 列表**。
6. **波末由你（或主会话）统一**：抽查 §6、`git diff --check`、`mkdocs build`；通过后勾选本波闸门。

### 8.3 波次总览

```text
W0 立规（可 2～3 并行） → 你验收
W1 定标样板（3 并行） → 你验收冻结样例
W2 入口与模块地图（4 并行） → 你抽查售前浅层+模块首页
W3 到货入库链（4 并行） → 你抽查
W4 库存/发料/主数据（4 并行） → 你抽查
W5 扩展业务模块（4 并行） → 你抽查
W6 平台与共享模型（4 并行） → 你抽查
W7 收尾与门禁（串行/少并行） → 关闭润色主线
```
