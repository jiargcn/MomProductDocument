# 正式业务文档标准模板（第一阶段）

> 状态：2026-07-22 按「字段行为模式 P1–P14」扩围修订；同日并入「润色」专节（§11）；原 2026-07-16 双样板结构与四层位置/拆分判定仍适用。
> 适用范围：DBC、WMS 及后续模块的模块首页、业务分组页、主文档和按需独立的维护与查询参考页。
> 首要目标：正式叶页写成给 **测试 + 实施 + 运维**（三者并列）用的「业务行为说明书」（见 §11）；运维含答疑、日常维护与排障，细节需求与测试、实施同级。顺带让业务人员能看懂、培训人员能讲清。不是把源码或数据库逐段翻译成页面正文。
> 字段语义写作细则见正式站 [`页面数据字典规范`](../../docs/02-业务模型/04-页面数据字典规范.md)；历史建设计划（已归档）见 [`字段业务语义建设计划`](../90-归档/01-历史总控过程/字段业务语义建设计划.md)；润色北极星与验收见 [`文档润色与可读性优化方案`](文档润色与可读性优化方案.md)。

## 1. 一套内容的四层位置

| 层次 | 面向谁 | 回答什么 | 放置位置 |
| --- | --- | --- | --- |
| 模块首页 | 新人、实施人员、跨模块协作人员 | 这个模块解决什么问题、与谁协作、先学什么。 | `docs/<模块>/index.md` |
| 业务分组页 | 使用者、培训人员 | 这一组业务包含什么、各页面怎样配合、学习顺序是什么。 | `docs/<模块>/<分组>/index.md` |
| 主文档 | 业务使用者、培训人员 | 为什么要做、怎样完成一次典型业务、出现异常怎么办、做完影响什么。 | 业务功能页或分组首页 |
| 维护与查询参考页（按需独立） | 日常维护、查询、测试人员 | 新增、编辑、导入、动作、查询、详情和联查怎样执行。 | 与主文档并列，命名为`<主文档名称>-维护与查询参考` |
| 内部证据与问题 | 文档维护、产品、研发人员 | 结论依据、字段技术名、源码差异、未解决问题。 | `project-docs/`，不作为培训正文链接 |

每个业务都必须覆盖维护与查询信息，但不必须拆成两份文档。主文档讲清为什么做、怎样完成一笔典型业务及其影响；维护与查询信息讲清怎样维护、怎样查询和怎样避免操作错误。简单业务将这些内容合并在主文档中；规则较多的业务再独立为`<主文档名称>-维护与查询参考`，不能以字段技术清单替代。

### 主文档与维护与查询参考的拆分判定

满足下表任意 **两项** 时，使用“主文档 + 维护与查询参考”；未满足时采用“单文档（维护与查询内容合并）”。判定结论必须登记在分组页或迁移清单中，避免按个人偏好拆页。

| 判定项 | 说明 |
| --- | --- |
| 导入或批量处理 | 有独立导入模板、错误回执、批量更新或批量执行规则。 |
| 状态与动作限制 | 有状态流转、动作前置条件、审批、撤回、关闭或不可逆操作。 |
| 多入口执行 | 同一业务存在 Web、PDA、线边端或其它终端入口，且规则或操作差异需要说明。 |
| 多对象追溯 | 需要串联三个及以上业务对象、单据或库存/质量结果才能完成查询。 |
| 内容密度 | 单页已超过约 1,500～2,000 字，或维护查询规则已明显打断业务主线。 |

物料、供应商、采购收货和库存管理作为已确认的双文档样板；仓库、库区、月台、班次等简单对象可先采用单文档，并在同页覆盖维护与查询内容。

## 2. 先定阅读路径，再建立完整大纲

正式页应优先满足下列阅读路径：

```text
我为什么需要它 → 我需要先准备什么 → 我如何完成一次业务 →
什么情况下能做什么 → 做完会影响什么 → 如何查询和处理异常
```

任何页面第一次建立时，必须一次性列出与页面类型相适应的完整章节。尚不能写实的内容使用**可分派的站位段落**（见[缺口标记写法规范](缺口标记写法规范.md)），不得只留空标题或空表。

```markdown
## 典型业务流程

!!! tip "📝 待补充"
    用一笔典型业务说明触发原因、参与角色、操作顺序和完成结果。

!!! example "📐 图示占位"
    一张“触发 → 处理 → 结果”的流程图；说明起点、终点、关键分支和图要回答的问题。

!!! example "📷 截图占位"
    给出需要准备的脱敏单据、物料、数量或状态，以及读者应看到的结果（也可用示例数据占位同级 example）。
```

站位段落至少要写清三件事：后续要补什么、为什么需要图/截图/示例、需要从何处取得资料。技术取证状态不属于站位内容。

### 既有页面迁移的强制规则

对已有正文进行重构时，**不得先清空页面再只留下占位**。应先逐段分类：

| 原有内容类型 | 正确去向 |
| --- | --- |
| 已能由当前菜单、源码、DDL 或测试环境支撑的业务结论 | 改写为正式页中的业务目的、流程、关键判断、查询、异常或限制。 |
| 字段技术名、表名、源码路径、接口调用、完整字段表、权限注解 | 按业务对象迁入 `project-docs/10-按需查阅/资料与证据/字段证据/`。 |
| 中文反译出的错误技术名、历史设计假设、互相冲突或未证实的结论 | 不写成当前规则；登记问题或写为明确的待确认事项。 |

占位只用于当前没有可靠资料、但已经知道后续需要补什么的部分。正式页至少应保留读者现在可依赖的业务内容；清除技术表达不能成为清空业务内容的理由。

```markdown
| --- | --- | --- |
| …… | …… | …… |

    模块与 DBC/WMS/其它上下游模块的业务关系图。

## 业务分组与学习顺序

| 建议顺序 | 业务分组 | 先学什么 | 学完能做什么 |
| --- | --- | --- | --- |

!!! tip "📝 待补充"
    前置主数据、权限、配置和上游单据。
```

## 4. 业务分组页模板

```markdown
| 页面 | 文档形态 | 已说明内容 | 后续需补 |
| --- | --- | --- | --- |
```

## 5. 主文档模板：把业务讲成一条完整的线

适用于主数据、配置、业务单据、申请、任务、记录和终端业务。以下章节按业务适用性选择，不适用时写清“不适用及原因”，不要机械堆表。

```markdown
# <功能或业务名称>

> 适用基线：……
> 阅读对象：……

## 业务目的与适用范围

!!! tip "📝 待补充"
    **开篇承诺（两句内）**：本页解决什么问题；读完能掌握什么（功能范围 / 逻辑 / 配置影响）。谁在什么场景使用、从哪里开始到哪里结束。

## 如何使用本组文档

| 你的目的 | 建议阅读 |
| --- | --- |
| 想理解业务主线 | 本页的流程、关键判断和异常处理。 |
| 正在具体维护、执行或查询 | <维护与查询参考页；如未拆页，则阅读本页的维护与查询章节>。 |
| 需要核对实现证据 | 由文档维护人员查内部底稿。 |

## 使用前准备

!!! tip "📝 待补充"
    前置数据、配置、权限、上游单据和现场条件。

## 一笔典型业务如何完成

!!! tip "📝 待补充"
    按时间顺序说明角色、操作和每一步结果。

!!! example "📐 图示占位"
    流程图；复杂业务应包括来源、任务、记录和下游影响。

!!! example "📝 示例数据占位"
    一笔正常业务的脱敏样例，包含关键数量/状态/关联号。

## 关键判断、角色与动作

!!! tip "📝 待补充"
    什么条件下能做什么、谁负责判断、差异或失败如何转向。业务逻辑要点（状态/对象/跨模块边界）写在这里。

## 配置如何起作用

!!! tip "📝 待补充"
    业务类型、单据设置、开关、规则、策略等如何改变流程/可选范围/现场行为。
    若本页无独立配置，写明「受某某通例/上游配置约束」。

## 建议验证点

!!! tip "📝 待补充"
    给测试/实施/运维的短列表（配置差异、状态门禁、异常、跨模块影响）；不是完整测试用例集。

| …… | P? | …… | …… | …… |

无行为相关字段的配置展示页，可写一句「本页无可维护行为字段，仅展示/跳转」并跳过本表。

## 做完后会影响什么

!!! tip "📝 待补充"

!!! tip "📝 待补充"

## 图示、截图与示例任务

!!! example "📷 截图占位"
    明确素材、目的、脱敏要求和资料来源。无图时可用写实示例（给定配置 → 期望行为）顶上。
```

### 主数据与配置页的补充重点

主文档中的“关键判断”至少覆盖：编码/名称识别、分类和单位、用途与管控、启停/替代/归档的影响。导入规则和列表细节优先放到维护与查询参考；未拆分的页面则在主文档中集中说明，主线部分只说明何时适合批量维护及其风险。配置页尤须写清「改哪个配置 → 行为如何变」。

### 申请、任务与记录页的补充重点

主文档先按“来源 → 申请 → 任务 → 现场执行 → 记录 → 下游结果”讲故事。公共概念只引用[申请、任务与记录模型](../../docs/02-业务模型/01-申请任务记录模型.md)，本页只说明该业务的来源、现场差异、库存/质量/接口影响和异常分支。

## 6. 维护与查询参考页模板：按需独立，让人能完成具体操作

当页面符合拆分判定时，建立维护与查询参考。该页只补“怎样做、怎样查、怎样定位”，不复述主文档的完整业务故事；即使当前资料不足，也要先建立完整大纲和可分派站位。文件名统一为`<主文档名称>-维护与查询参考.md`。未拆分时，将下列适用章节合并到主文档的“查询、追溯与快速跳转”“关键判断”或独立的“维护与查询说明”章节。

### 维护与查询参考页的表格规则

| 需要说明什么 | 推荐写法 | 不要写成什么 |
| --- | --- | --- |
| 关键字段业务语义 | 按 P1–P14 归类；业务作用、谁维护、约束、影响、错选风险。 | 全量技术字段清单、DDL/DO/DTO 映射。 |
| 选择器（P2/P3） | 选择对象、**可选范围**、依赖、带回/锁定、选不到原因。 | 只写“弹窗选择××”；接口路径和请求参数。 |
| 字典/开关（P1） | 取值 → 功能影响矩阵。 | “来自字典”而无行为落点。 |
| 联动/回填（P4/P5） | 上游变化如何处理下游；回填可改/锁定。 | 控件实现细节。 |
| 系统限制 | 只写会影响录入/导入的长度、格式、重复或数值范围。 | 字段类型/数据库列定义的完整翻译。 |
| 状态与动作（P9） | 当前能做什么、先检查什么、结果是什么。 | 未验证的固定状态机。 |
| 列表和详情 | 推荐字段顺序、筛选、分组、联查目的。 | 为覆盖而展示所有字段。 |

## 7. 图示、截图与示例的最低要求

| 内容 | 何时需要 | 站位必须写清 |
| --- | --- | --- |
| 流程图 | 多角色、多单据或跨模块流转。 | 起点、终点、关键分支和读者要理解的关系。 |
| 状态图 | 动作受状态限制。 | 状态、允许动作、回退/关闭分支及验证资料。 |
| 对象关系图 | 三个以上业务对象需要串联。 | 业务对象和关系，不画表名、字段和服务调用。 |
| 页面截图 | 需要解释表单、列表、按钮、分组或异常提示。 | 页面区域、要标出的内容、脱敏要求。 |
| 示例数据 | 数量、状态、关联关系、导入或差异规则复杂。 | 场景、关键数据、预期结果和查询入口。 |

## 8. 写作与验收流程

1. 先判断页面类型，并按拆分判定选择“单文档（维护与查询内容合并）”或“主文档 + 维护与查询参考”。
2. 一次性建立完整大纲和站位；先不填技术字段。
3. 用一笔典型业务写出主线，再补关键判断、**配置如何起作用**、**建议验证点**、查询和异常（见 §11.2）。
4. **字段语义：** 列出行为相关字段 → 按 P1–P14 归类 → 主文档关键角色表 → 参考页语义总表 + 实际需要的专项表。
5. 仅把确实影响操作的字段和限制写进维护与查询参考；其余证据留在 `project-docs/`。
6. 检查维护与查询信息是否完整：成对页面应互补，单文档应在同页提供可操作的维护与查询说明。
7. **叶页润色：** 按 §11.3 单页验收清单自检（测试/实施/运维同级，见 §11.3）；通过后再由业务、培训、文档维护审阅，进入下一批。

- 是否集中说明当前限制，而没有打断主流程？

## 10. 禁止事项

- 不把 DDL、DO、DTO、VO、Controller、接口调用链当作业务正文。
- 不以“代码已证实”“DDL/DO 已证实”“当前实现事实卡”等技术标签开篇。
- 不用空表、空标题代替站位段落；句中缺口用徽章，整段用 admonition。
- 不为了覆盖字段而打断一条完整业务流程。
- 不把历史设计、个人推断或未验证实现写成用户可依赖的规则。

## 11. 润色约定（测试 / 实施 / 运维北极星）

> 本节为增量约定，**不推翻** §1 四层位置、拆分判定、§3～§6 页面骨架与字段语义写法。完整方案见 [文档润色与可读性优化方案](文档润色与可读性优化方案.md) §0～§7。叶页润色与抽查一律按本节执行。

### 11.1 北极星（唯一）

把正式业务叶页写成给 **测试 + 实施 + 运维（P1，三者并列）** 用的「业务行为说明书」——功能范围清楚、逻辑可讲、配置可落地、验证可开单；运维同样能据此**答疑、日常维护与排障**（配置/状态/跨模块影响），对系统细节的了解需求与测试、实施同级。**禁止**把运维降级为「只看平台与集成」或仅部署监控。

| 角色 | 地位 | 对模板的含义 |
| --- | --- | --- |
| **测试 / 实施 / 运维（P1）** | **唯一润色北极星（三者并列）** | 叶页验收、文风密度、章节义务全部按他们定；运维含答疑/维护/排障，不仅是部署 |
| **新开发（P2）** | 顺带受益 | 与 P1 同向；不另设文风 KPI |
| **售前 / 销售（P3）** | 只做浅层入口 | 首页 + 总体框架 + 模块「解决什么问题」；**叶页不为售前降维** |
| **研发自身（P4）** | **质量约束，非文风 KPI** | 只写已确认业务事实；冲突进总账；技术对照在证据层。不以「对照源码」为润色成功标准 |

**可读性定义：** 结构清楚、可扫读、少废话、有主线 → 读者**愿意读完**。不追求故事腔、趣味化或「有趣」KPI。

**主文档密度：** 允许为讲清配置影响、状态门禁、跨模块影响与验证点而「干一点、密一点」；完整字段表仍下沉维护参考。对 P1，「清楚、能用」优先于「轻薄好看」。

### 11.2 主文档必须有的章节义务

对齐润色方案 §5.1。与 §5 骨架并用；不适用时写清「不适用及原因」，不得整段省略「配置如何起作用」「建议验证点」。

1. **开篇承诺**（两句内）：本页解决什么问题；读完能掌握什么（功能范围 / 逻辑 / 配置影响）。
2. **如何使用本组文档**：分流主文档 vs 维护参考（给测试/实施/运维）。
3. **一笔典型业务**（文字 ± mermaid）：触发 → 处理 → 结果；含关键分支。
4. **使用前准备**、**业务逻辑要点**（状态/对象/跨模块边界）、**关键判断或规则**。
5. **配置如何起作用**：业务类型、单据设置、开关、规则、策略等；若无独立配置，写明「受某某通例/上游配置约束」。
6. **建议验证点**（短列表；不是完整测试用例集）。
7. **做完影响什么**、**异常与查询入口**（可链维护参考；便于运维答疑与排障）。
8. **关键字段业务角色**（仅主线；建议约 8～12 行内，超出下沉维护参考）。允许为讲清配置/门禁而略密，但不得变成字段全家福。

**主文档禁止 / 限制（对齐方案 §5.2）：**

- 禁止 DDL / DO / DTO / VO / 源码路径 / 权限注解原文 / 未证实技术名。
- 禁止以 P1–P14 编号作为读者可见标题。
- 禁止开篇即大表；先主线与逻辑，后表。
- 禁止把问题总账全文铺进正文。
- 禁止为讨好售前而删掉配置/验证要点。
- 禁止为「方便对照代码」而写入技术证据。
- 禁止把运维读者指引成「只需看基础设施/部署页」而跳过业务叶页深读。

### 11.3 单页润色验收清单（叶页；P1 视角——测试 / 实施 / 运维同级）

叶页润色完成时须同时满足（抽查问测试 / 实施 / 运维，三者同级）：

- [ ] 开篇承诺清晰（问题 + 读完能掌握什么）
- [ ] 有「如何使用本组文档」分流
- [ ] 有一笔典型业务主线（文字或图）
- [ ] 有「配置如何起作用」（或明确无独立配置及所受约束）
- [ ] 有短「建议验证点」
- [ ] 有准备 / 逻辑或判断 / 影响 / 异常中的可执行说明
- [ ] 关键字段表服务主线，未膨胀成字典页
- [ ] 术语首次有场景义或共享链接
- [ ] `❓` / GAP 不打断主线（短标或文末）；**未借润色删掉或改没**既有 `❓`/待确认/待核验/待完善及站位高亮
- [ ] 无技术证据腔
- [ ] 上下游至少各有一个有用链接（若存在协作）
- [ ] **抽查**：测试能据此列场景；实施能讲配置影响；运维能据此答疑/维护/排障（配置·状态·跨模块影响）
- [ ] **否决项**：未为售前删深度；未为对照代码加技术细节；未把运维降为「只看平台」
- [ ] **否决项（结构 / 方案 §6、§7.1）：** 未用短地图或新骨架整页覆盖；未删除既有章节/站位；确认为以旧版为底的增量合并
- [ ] **否决项（站位 / 待确认 · 一票否决）：** 未误删截图/图示/示例数据站位、`待补充的图示与示例`、待确认/待核验/待完善及 [缺口标记写法规范](缺口标记写法规范.md) 约定的醒目标记（含 `!!! example "📷 截图占位"`、`📝 示例数据占位`、`❓` 等）

> **W0-A 说明：** 本任务只改标准模板、不改业务页；将上表写入模板即完成「§6 并入模板」义务。后续内容波改叶页时，按本清单逐页勾选。

### 11.4 反模式（出现则退回）

对齐润色方案 §7 / **§7.1**（结构硬禁令 + 禁删清单）：

1. 开篇先贴字段全家福或「行为模式清单」。
2. 用「该字段参与上游关联」代替「选错/配错会导致什么」。
3. 把 DDL、类名、API path、权限注解写进正式页。
4. 正文变成 GAP/待确认清单（**≠** 删掉站位或待确认标记；标记须保留，只是不要让全文变成缺口堆砌）。
5. 主文档与维护参考重复且无分流。
6. 售前路径链进维护参考或字段长表。
7. 叶页为「售前也好懂」而砍掉配置/验证要点。
8. 为润色清空已证实业务结论（只许改写表达）。
9. 无依据脑补配置影响或过滤条件。
10. 把「有趣/故事感」当成验收通过条件。
11. **润色 ≠ 重构：** 禁止用短地图/新骨架整页覆盖旧结构；禁止删除既有章节（只许改写增强或增量加节）。结构禁删含：学习/维护/操作顺序、谁会使用、关键字段业务角色、业务目的、如何使用本组文档、常见问题、关键对象/边界，以及页内已有的同类导航/范围说明——只许以旧版为底增量加节或改写增强（见方案 §7.1；W2 短地图波次为失败教训）。
12. **误删站位 / 待确认高亮 = 一票否决（禁删清单）：** 下列带 emoji 或高亮内容**必须保留**，禁止被「润色 / 优化 / 去工程腔 / 去 emoji」删掉或改没：
    - 截图占位（含 `!!! example "📷 截图占位"` 等）
    - 图示占位（含 `!!! example "📐 图示占位"` 等）
    - 示例数据占位（含 `📝 示例数据占位` 等）
    - 待补充站位与章节（含 `!!! tip "📝 待补充"`、`待补充的图示与示例`）
    - 待确认 / 待核验 / 待完善及同类站位（含 `❓ 待确认`、`🔍 待核验`）
    - [缺口标记写法规范](缺口标记写法规范.md) 约定的醒目标记（若有 `❓` 业务待确认，保留；**不要把「待确认」整类清掉**）
    
    **口径：** 禁止的是「售前勿进」「本刀」「北极星」等过程黑话进正式页；**不是**禁止「待确认 / 截图占位」。若与缺口规范写法细节冲突，**以「必须保留」本裁决为准**。
