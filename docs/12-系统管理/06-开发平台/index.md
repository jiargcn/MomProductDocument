# 开发平台

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；开发与配置管理员（顺带）。业务用户一般不直接依赖本页完成现场作业。

## 业务目的与适用范围

业务异常发生时，别急着怀疑「开发平台又出问题了」——多数时候问题出在业务规则或权限，不在这里的配置。开发平台承接面向扩展与配置的支撑能力（代码生成、系统参数配置、报表设计等），**不是**仓储/生产业务模块。

读完本页，应能：判断「配置改了不生效」该先查缓存/租户/键名，还是回业务规则与权限；并知道代码生成、报表、文件、字典等应落到哪一页，而不是把业务规则写进本页。

## 当前入口

| 入口 | 作用 | 风险提示 |
| --- | --- | --- |
| 代码生成 | 按表生成前后端骨架，加速开发。 | 仅用于受控开发环境；勿在生产直接生成覆盖业务定制。 |

| 字典 / 菜单 / 权限 | System 主数据与授权。 | 见系统管理对应页，不在本页展开字段。 |

!!! example "写实示例：给定配置 → 期望行为"
    **给定：** 在测试环境修改一条基础设施参数键值（可见参数）；目标租户正确；变更已保存。
    **期望：** 依赖该参数的页面在刷新缓存或重新登录后按新值生效。若无效：先核对键名、可见性、租户，再怀疑业务规则。代码生成仅在受控开发环境执行；生产直接生成覆盖定制属禁止路径。

### 建议验证点

- 配置管理入口仅对具备菜单权限的角色可见（RBAC）。
- 改可见参数后：刷新/重登后行为变化可观察；改错租户则目标租户无变化。
- 代码生成：测试/开发环境可进入；生产环境按实施约定禁止或严控。
- ❓ 低代码表单/流程设计器若菜单存在，启用范围待环境确认——未证实勿写入必测。
- 业务异常归因：先分清业务规则、权限、参数，再动开发平台。

## 编写边界（必须遵守）

内容写错地方是文档维护最常见的坑，下表是三条硬边界：

| 内容 | 应写在哪里 |
| --- | --- |
| 平台提供什么配置能力 | 本页 / 系统管理 |
| 配置如何监控、审计、发布 | 基础设施（日志、定时任务、数据交换等） |
| 业务如何使用配置结果 | 对应业务模块页面 |

## 关键判断

| 判断点 | 应先确认什么 |
| --- | --- |
| 配置改了不生效 | 是否需刷新缓存/重新登录、是否改错租户。 |
| 代码生成覆盖定制 | 是否在错误环境执行；对比版本管理。 |
| 业务异常被归因到“开发平台” | 先分清是业务规则、权限还是参数配置。 |

## 当前限制与待确认事项

- ❓ 低代码表单/流程设计器若菜单存在，启用范围待环境确认（审批引擎边界见岗位/BPM 说明）；
- ❓ 配置发布、灰度与回滚的标准作业程序待运维补齐；
- ❓ 开发平台变更的强制审计字段矩阵待抽样。
