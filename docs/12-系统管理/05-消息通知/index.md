# 消息通知

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；需要日常查看/发布通知的业务管理员（顺带）。

## 业务目的与适用范围

公告发了没人看、站内信收不到——先别怀疑功能坏了，本页只管「用户能直接用的入口」这一层：通知公告、我的消息/消息记录、站内信模板，以及环境启用时的钉钉模板与记录。

读完本页，应能：判断问题该在本页查用户入口与发送记录，还是切到基础设施查邮件/短信等渠道配置；并明确站内信已读 ≠ 业务任务完成。平台渠道分层、跨模块待办见：[消息、通知与待办](../../03-基础设施/06-消息、通知与待办.md)。

### 建议验证点

- 有菜单的用户能打开「我的消息」；无菜单则不可见（RBAC）。
- 发布公告后：目标用户可见；停用/未发布则不可当已生效公告验收。
- 收不到站内信：先查消息记录是否生成、接收用户是否正确，再查模板与渠道。
- 邮件/短信收不到：不要只在本页找——回基础设施消息页确认环境是否启用该渠道。
- ❓ 业务事件 → 自动发信矩阵未建全；未证实触发勿写成必测。

## 与业务待办的边界

把消息中心当成待办中心是最常见的误用，记住三条：

- 站内信已读 ≠ 业务任务完成。
- 仓储任务领取、质检通知、ANDON 响应等以对应业务列表为准。
- 不要把 System 消息中心培训成唯一待办中心。

## 常见问题

| 情况 | 建议处理 |
| --- | --- |
| 收不到站内信 | 查消息记录是否生成、用户是否正确、模板是否发送。 |
| 公告发了没人看 | 确认公告状态/范围；重要事项仍需业务待办闭环。 |
| 找不到邮件短信配置 | 回基础设施消息页看渠道分层，并确认环境是否启用。 |
