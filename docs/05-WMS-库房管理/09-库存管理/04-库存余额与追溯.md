# 库存余额与追溯

> 适用基线：测试环境 / `dev` 分支 / 2026-07-15。具体查询操作见[库存管理-维护与查询参考](01-库存管理-维护与查询参考.md)。

## 这项查询解决什么问题

库存余额展示当前可查询到的库存结果。查询余额时必须把物料、地点、库存状态、批次、托盘和包装作为一个整体理解，不能把同一物料的不同追溯单元简单相加后直接用于业务判断。

## 字段行为要点（P3 / P8 / P9）

模式归类：**P3（按完整业务键理解当前结果）**、**P8（批次/托盘/包装等采集维度的落账结果）**、**P1/P9（状态/冻结影响是否可用）**。
通例见[库存管理精度与唯一粒度](../../02-业务模型/08-库存管理精度与唯一粒度.md)。完整总表见[库存管理-维护与查询参考](01-库存管理-维护与查询参考.md)「字段说明」。

| 要点 | 说明 |
| --- | --- |
| 业务键 | 批次、托盘、包装、物料、库存状态、库位共同决定“是不是同一笔库存” |
| 查询顺序 | 物料+库位+状态 → 批次/托盘/包装 → 数量 → 最近事务 |
| 门禁 | 余额页不作日常改账入口（`GAP-012`）；差异走盘点/退货/库内等源业务 |
| 反查风险 | 详情反查事务时过滤可能未带齐托盘/状态/库位等（`GAP-022`）；须人工核对 |
| 唯一约束 | 业务键已确认，DDL 等价唯一约束仍待核验（`GAP-005`） |

## 如何从余额追溯原因

```mermaid
flowchart RL
    A["当前库存余额"] --> B["最近库存事务"]
    B --> C["业务记录"]
    C --> D["任务与来源单据"]
```

当余额异常时，先确认查询维度，再追溯最近事务和业务记录；不要直接修改余额来掩盖来源业务、状态或地点错误。

## 查询与维护说明

| 目的 | 首要条件 | 详情应分组 | 快速跳转 |
| --- | --- | --- | --- |
| 查现有库存 | 物料、库位、库存状态。 | 识别与数量、地点与状态、批次与包装、来源与最近变化。 | 库位、库存事务、业务记录。 |
| 查可用范围 | 库存状态、冻结信息、地点。 | 可用/限制信息、质量或冻结线索。 | 质量、冻结或异常处理（待补）。 |
| 查差异原因 | 批次、托盘、包装和最近事务。 | 追溯维度、数量变化、来源与时间。 | 事务、收货/退货/上架记录。 |

当前余额业务粒度与查询匹配仍存在唯一性差距（`GAP-005`）；从余额详情反查事务时，过滤条件可能未带齐托盘、库存状态、库位等维度（`GAP-022`）。在解决前，查询和业务判断必须完整带入追溯维度，并对详情中的事务列表做人工核对。

!!! example "📝 示例数据占位"
    同一物料因库位、批次、包装和库存状态不同形成多条余额，并从其中一条反查收货/上架记录。
