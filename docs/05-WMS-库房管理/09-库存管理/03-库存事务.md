# 库存事务

> 适用基线：测试环境 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）。
> 具体查询操作见[库存管理-维护与查询参考](01-库存管理-维护与查询参考.md)。

## 这项查询解决什么问题

库存事务记录**一次已经发生的库存变化**。它是解释「为什么数量、地点或库存状态变了」的主要入口，而不是日常人员绕开来源业务直接改库存的工具。读完本页，应能按业务记录号或事务号定位一次变动，并判断下一步该看余额还是该回业务记录。

**与另外两层的边界：**

| 本页（事务） | 不是本页 |
| --- | --- |
| 已发生：某次业务造成的变动事实 | [库存预期](02-库存预期.md)：仍待完成的入/出预期 |
| 回答「为什么变了、来自哪条记录」 | [库存余额与追溯](04-库存余额与追溯.md)：变动之后的**当前结果** |
| 由源业务记录生成（培训主路径） | 勿把本页当手工造账入口（`GAP-012`） |

三类对象总览见[库存管理](index.md)。排障习惯路径：**余额异常 → 最近事务 → 业务记录**；或 **业务记录 → 事务 → 余额** 核对是否过账成功。

## 与入库链的关系

采购收货记录已确认会形成库存事务；采购退货和采购上架完成后应通过事务验证其库存结果。上架通常更关注地点或状态变化，退货更关注可退对象与余额扣减，实际事务口径仍需分别核对。

!!! example "写实示例：给定业务 → 期望行为"
    **给定：** 收货记录 RCV-2001 实收物料 A 98 件已提交；随后上架记录 PUT-3001 将其中 60 件从上架待定位移到库位 B-01。
    **期望：**

    1. 按收货记录号能查到对应入库类事务（数量口径与实收一致，拒收不计入正常实收）。
    2. 按上架记录号能查到地点/状态变化类事务；余额在 B-01 的增加应能追溯到该事务。
    3. 培训路径始终是「源业务 → 记录 → 事务」；不要为「把数改对」而在事务页手工旁路写入（人工入口是否开放见 `GAP-012`）。

### 建议验证点

- 收货/退货/上架完成后，分别用业务记录号定位事务，再联查余额是否一致。
- 只知物料不知记录时：同时带地点、状态、批次/包装与时间窗缩小范围。
- 导出等动作的权限以实测为准（`GAP-021`，并挂 `GAP-014`）。

## 查询时看什么

| 要点 | 说明 |
| --- | --- |
| 主键追溯 | 优先用业务记录号 / 事务号，再带物料与时间窗 |
| 粒度字段 | 物料、库位、库存状态、批次、包装等应与余额口径一致 |
| 动作边界 | 培训路径：源业务 → 记录 → 事务；人工创建是否开放见 `GAP-012` |
| 导出等动作 | 前端可有权限码，后端强制鉴权证据不足（`GAP-021`，并挂 `GAP-014`） |

完整字段语义见[维护与查询参考](01-库存管理-维护与查询参考.md)。

## 查询与维护说明

| 场景 | 建议优先显示/筛选 | 联查路径 |
| --- | --- | --- |
| 解释一次变化 | 事务号、业务类型、事务类型、记录号、物料、数量、发生时间 | 业务记录、来源任务、来源单据 |
| 查地点或状态变化 | 物料、库位、库存状态、批次/包装、数量 | 库存余额、库位资料 |
| 发现异常 | 处理人、时间、来源记录和关联事务 | 记录明细、接口/操作日志（待补） |

当前已确认存在创建库存事务和创建转运日志的服务入口，以及分页、高级查询与导出能力；正式培训应始终优先从来源业务形成事务。人工创建是否开放、可修改哪些信息和权限边界待实测。
