# 采购退货-维护与查询参考

> 本页是[采购退货](index.md)的操作参考；它与采购收货共用“申请—任务—记录”的业务模型，但退货来源、可退范围和库存扣减必须单独核对。

## 各阶段应维护什么

| 阶段 | 应完成的业务信息 | 不应绕过的限制 |
| --- | --- | --- |
| 退货申请 | 来源收货、供应商、物料、可退数量、批次/包装、退货原因和交接要求。 | 未确认来源或可退范围时不得直接形成退货结果。 |
| 退货任务 | 执行人、执行地点、待退实物与交接安排。 | 仅处理已进入可执行状态的任务；具体状态待实测。 |
| 退货记录 | 实退数量、批次/包装、交接结果、异常原因。 | 已完成记录的修改、撤销与反向库存影响需走受控处理。 |
| 撤销记录 | 已撤销的退货处理及原因。 | 不能把撤销当作删除历史；库存和外部协同结果必须联查。 |

## 导入与批量处理

当前后端可见多类退货申请导入对象，覆盖常规、跨月、备件及制造订单类型等场景；具体模板、可导入字段、更新策略和前置校验尚未完成端到端验证。

导入前至少核对：来源可追溯、供应商/物料有效、数量不超可退范围、批次/包装与当前库存一致、退货原因完整。系统内部编号、历史状态和审计信息不应作为业务模板填充项。
