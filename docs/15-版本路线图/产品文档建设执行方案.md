# MOM 产品文档建设执行方案

> 状态：已确认，待执行  
> 版本：1.5
> 更新日期：2026-07-15  
> 分析基线：测试环境 / dev 分支 / 2026-07-15  
> 核心目标：把产品文档建设成真正可用于培训、业务查询和产品改进的工作资料。

## 1. 建设定位

本次文档完善不以“页数多、字段多、图表多”为目标，而以“真的有用”为第一判断标准。

完成后的文档应同时具备两种用途：

1. **培训材料**：新人、实施人员或跨模块人员能够理解业务背景、角色分工、操作过程以及上下游影响。
2. **业务字典**：产品、开发、测试和实施人员能够快速查到模型、单据、状态、字段、规则、数据来源、权限和接口关系。

一篇文档是否合格，不看它写了多少内容，而看它能否回答以下问题：

- 这项功能解决什么业务问题？
- 谁在什么场景下使用？
- 使用前需要准备什么，完成后会产生什么结果？
- 页面上的关键字段是什么意思，数据从哪里来？
- 哪些内容可以新增、编辑、导入，限制是什么？
- 单据会生成什么任务、记录、库存变化或外部系统数据？
- 出现异常时如何判断和处理？
- 当前系统已经做到什么，哪些地方仍需完善？

## 2. 主要使用者与使用方式

| 使用者 | 主要需求 | 文档应提供的内容 |
|--------|----------|------------------|
| 新员工 | 快速理解系统和业务 | 场景说明、角色、流程、示例、常见问题 |
| 业务人员 | 确认规则和数据含义 | 业务术语、单据关系、字段含义、状态和限制 |
| 实施顾问 | 配置、培训和问题定位 | 前置数据、配置顺序、权限、导入、异常处理 |
| 产品经理 | 梳理现状和规划改进 | 当前实现、目标要求、产品差距和待决策项 |
| 开发人员 | 理解业务意图和影响范围 | 模型、状态、业务规则、接口、数据来源 |
| 测试人员 | 设计测试场景 | 前置条件、操作限制、状态转换、异常和边界条件 |
| 运维人员 | 定位系统和数据问题 | 日志、任务、接口、重试、审计和排查入口 |

## 3. 编写原则

### 3.1 业务场景优先

先说明“为什么做、什么时候做、谁来做”，再介绍页面和字段。不能把数据库字段表直接当作产品文档。

### 3.2 先讲主线，再讲例外

培训内容先给出最常见的标准流程，再补充部分完成、撤销、超量、短缺、重复提交等异常场景。

### 3.3 一个概念只完整解释一次

“申请—任务—记录”、库存事务、库存余额、预计入、预计出、RBAC 等共享模型集中说明。具体业务页只引用共享模型并描述本业务的差异。

### 3.4 业务语言与技术标识并存

正文以业务人员能理解的中文表达为主；字段名、CODE、API、表名和错误信息作为查询依据保留，但不主导阅读顺序。

### 3.5 当前实现与目标设计分开

不能把产品设想写成当前功能，也不能因为当前没有实现就不写合理的目标要求。统一使用以下标识：

| 标识 | 含义 |
|------|------|
| 当前已确认 | 已通过页面、代码、接口、数据库或运行数据验证 |
| 当前待确认 | 有初步依据，但证据不完整 |
| 目标设计 | 产品希望达到的业务要求 |
| 产品差距 | 当前实现与目标设计不一致 |
| 待决策 | 需要产品或业务负责人选择方案 |

### 3.6 技术字段名必须以证据为准

中文标签只用于解释业务含义，不能反向推导技术字段名。字段技术名必须按“DDL/实际表结构 → 后端 DO/Entity → API DTO/VO → 前端类型/请求参数”的证据链确认；发生冲突时登记为待确认，不自行翻译或猜测。详见[字段技术名真实性校正任务清单](字段技术名真实性校正任务清单.md)和[字段技术名问题初扫清单](字段技术名问题初扫清单.md)。

### 3.7 示例必须能帮助理解

优先使用贯穿场景的示例数据，例如“一张采购订单如何形成收货申请、任务、记录和库存变化”。截图只用于解释关键操作，不为每个字段机械截图。

### 3.8 不追求无差别的完整

优先写高频、高风险、跨模块、容易误解的内容。很少使用且没有特殊规则的字段可以简要记录，不应让大量低价值信息淹没关键规则。

## 4. 目标文档体系

文档按“系统认知—共享模型—模块培训—页面字典—产品改进”五层组织。

| 层级 | 主要内容 | 主要用途 |
|------|----------|----------|
| 系统认知 | 产品边界、能力地图、角色、上下游系统 | 入门培训 |
| 共享模型 | 通用单据、申请任务记录、库存、权限、集成 | 统一业务语言 |
| 模块培训 | 模块定位、角色、主流程、前置数据、典型场景 | 岗位培训 |
| 页面字典 | 字段、操作、查询、导入、详情、接口 | 日常查询和测试依据 |
| 产品改进 | 差距、待决策、建议和版本计划 | 产品规划 |

### 4.1 两种阅读入口

每个模块同时提供两种入口：

- **我要学习**：按业务场景和操作顺序阅读，例如“如何完成一次采购收货”。
- **我要查询**：按页面、单据、字段、状态或术语快速定位。

模块 `index.md` 应同时给出“建议学习顺序”和“业务字典索引”。

## 5. 系统级文档规划

### 5.1 总体框架

总体框架重点回答：系统服务谁、覆盖哪些业务、与 ERP/WCS/设备等系统如何分工。

需要补充：

- 产品定位和业务边界。
- 系统能力地图。
- 典型用户和岗位。
- 模块职责及数据归属。
- 上下游系统和关键数据流。
- 按角色划分的学习路径。

### 5.2 共享业务模型

`02-业务模型` 应优先建设以下内容：

1. 模型分类及设计目的。
2. 通用单据结构。
3. “申请—任务—记录”模型。
4. 单据状态、动作和生命周期。
5. 库存余额、冻结量、可用量和库存事务。
6. 预计入、预计出及其消耗和关闭规则。
7. 单据与库存数据的挂接及过账机制。
8. 第三方单据、外部 ID、同步状态和接口挂接。
9. 撤销、冲销、补偿、幂等和重复处理。
10. 审计、追溯和数据血缘。

这些页面既要解释设计目的，也要提供可查询的关系表、状态表和挂接字段表。

### 5.3 System：权限与系统治理

System 文档不能停留在用户、角色和菜单名称上，应从实际管理任务出发：

- 如何创建用户并赋予可工作的权限。
- 角色、岗位和组织之间是什么关系。
- 菜单、按钮、API、数据和字段权限如何生效。
- 用户能查看哪些工厂、仓库、车间和组织的数据。
- 谁能够批准、驳回、撤销或执行某类业务。
- 金额、数量、业务类型等条件是否影响决策权限。
- 如何处理临时授权、代理、离职和权限回收。
- 如何查询登录、授权、操作和数据变更记录。

RBAC 页面至少提供：角色定义、资源与动作、数据范围、典型授权示例、决策权限矩阵、常见配置错误和审计方法。

### 5.4 公共能力

公共能力按“谁使用、何时触发、如何配置、失败后如何处理”编写：

- 标签、条码和标签模板。
- 报表、打印模板、打印任务和打印记录。
- 看板、指标、组件、刷新和数据权限。
- 通用查询、保存查询、导出和大数据量处理。
- 文档、文件、附件、版本和预览。
- 消息、通知和订阅。
- 导入导出中心及处理日志。

每项公共能力应列出被哪些业务模块使用，以及业务页面如何调用或挂接。

### 5.5 Infrastructure：运行和问题定位

基础设施文档主要服务架构、开发、测试和运维，不强迫普通业务人员阅读。重点包括：

- 部署拓扑、环境和服务清单。
- 网关、服务调用、消息和任务调度。
- 数据库、缓存、对象存储和搜索。
- 应用日志、Trace、Metric、告警和审计日志的边界。
- 安全、加密、脱敏、数据保留和备份恢复。
- 接口限流、重试、幂等、死信和补偿。
- 常见问题的排查入口和判断顺序。

## 6. 模块级培训页面模板

每个业务模块的概述页应能独立完成一次模块入门培训。

建议章节如下：

1. 模块解决什么问题。
2. 谁使用该模块。
3. 使用前需要哪些主数据和配置。
4. 模块包含哪些业务场景。
5. 一条最典型的端到端流程。
6. 核心单据和模型关系。
7. 与其他模块和第三方系统的关系。
8. 常见异常和处理入口。
9. 推荐学习顺序。
10. 页面、单据和术语索引。

复杂模块可以把流程、模型和接口拆成独立页面；简单模块优先保持一个清晰的概述页，避免为了结构而拆页。

## 7. 页面级业务字典模板

### 7.1 主数据页面

主数据页面按实际维护任务组织：

1. 业务用途和被哪些业务引用。
2. 数据维护责任人和前置数据。
3. 新增步骤及关键约束。
4. 选择器的数据源、筛选条件、排序和权限范围。
5. 可编辑与不可编辑字段，以及状态和引用限制。
6. 启用、停用、删除和归档规则。
7. 导入模板字段、系统填充字段和校验规则。
8. 列表字段、顺序、默认排序和常用筛选。
9. 详情分组和快速跳转。
10. 常见错误、示例和产品差距。

字段信息拆成几张可读的小表，不使用一张超宽表承载全部信息。标准写法见[实体说明页标准模板](实体说明页标准模板.md)。

- 基础定义表：字段、含义、类型、长度、必填、默认值。
- 维护规则表：新增、编辑、系统生成和状态限制。
- 选择器表：数据源、筛选条件、展示字段和带回字段。
- 导入规则表：是否导入、模板列、校验和错误处理。
- 展示规划表：列表顺序、详情分组和跳转目标。

### 7.2 申请、任务和记录页面

按一条完整业务链说明：

- 业务触发来源和前置条件。
- 申请、任务、记录各自保存什么信息。
- 谁可以创建、审批、领取、执行、撤销和查看。
- 状态与允许动作。
- 一张申请如何拆分任务，一项任务如何形成记录。
- 哪个时点生成预计量、冻结量、库存事务和余额变化。
- 来源单据、后续单据和外部系统单据如何挂接。
- 部分完成、数量差异、取消、冲销和重复执行如何处理。
- 列表、详情分组和联查入口。

页面中应提供一个带具体数量的完整示例，帮助培训人员把模型与实际业务对应起来。

### 7.3 查询、记录和报表页面

查询类页面重点回答“怎样最快找到正确的数据”：

- 数据来源、统计口径和更新频率。
- 推荐查询条件和条件之间的关系。
- 列表显示字段、字段顺序、默认排序和汇总规则。
- 状态、数量和金额的计算口径。
- 详情页分组。
- 可跳转的来源单据、任务、记录、库存事务或外部单据。
- 导出范围、权限和大数据量限制。
- 常见的“为什么查不到”问题。

### 7.4 公共能力和系统配置页面

除配置字段外，还应说明：

- 该配置会影响哪些页面和业务。
- 修改后何时生效。
- 是否影响历史数据。
- 错误配置会造成什么后果。
- 如何验证配置已经生效。
- 谁可以修改，是否需要审计或复核。

## 8. 产品差距的记录方式

产品差距不能混在正文中反复打断培训阅读。每页末尾设置独立的“产品观察”章节，记录：

| 项目 | 内容 |
|------|------|
| 差距编号 | 例如 `GAP-WMS-RECEIVING-001` |
| 当前表现 | 已验证的当前行为 |
| 业务影响 | 对效率、准确性、培训或风险的影响 |
| 目标要求 | 希望达到的业务效果 |
| 建议方向 | 可选的改进思路 |
| 优先级 | P0/P1/P2 |
| 状态 | 待确认/已确认/已规划/已完成 |

跨页面差距汇总到版本路线图中，作为后续产品迭代输入。

## 9. 优先级规则

不按菜单顺序机械推进，按业务价值排序：

### P0：先完成

- 系统边界和能力地图。
- “申请—任务—记录”模型。
- 库存事务、余额、预计入和预计出。
- RBAC、数据权限和决策权限。
- 高频主数据及其维护规则。
- 采购收货、发料、生产收料、销售出库等主流程。
- 标签、打印、查询、日志等跨业务公共能力。

### P1：第二批

- 其他常规业务单据链。
- 质量、生产、设备、供应链等模块核心页面。
- 主要查询、报表、详情联查和导入规则。

### P2：持续补充

- 低频配置和长尾页面。
- 纯技术内部信息。
- 不影响理解和操作的次要字段。

## 10. 执行阶段

### 阶段 0：确认标准

确认本方案、页面模板、证据标识、差距编号和验收方法。

### 阶段 1：建立权威页面总账

使用菜单清单和前端路由建立“菜单—路由—组件—权限码—文档”映射，并为每个页面标记类型、优先级、负责人和资料状态。

### 阶段 2：完成共享模型和系统底座

先完成业务模型、RBAC、公共能力和基础设施框架，使后续页面有统一引用基础。

### 阶段 2.5：字段技术名真实性校正

在大规模修订叶页字段表、导入规则和详情字段前，先执行字段技术名真实性校正：建立字段映射表、校正纵向试点、清点既有错误字段并形成防回归规则。该阶段是页面字典推广的 P0 前置条件。

### 阶段 3：完成一个纵向试点

建议试点范围：

- 主数据：物料、仓库、库位。
- 业务链：采购收货申请—任务—记录。
- 库存链：预计入—库存事务—库存余额。
- 权限链：用户—角色—数据范围—业务动作。

试点必须同时产出培训内容、业务字典和产品差距，以验证模板是否真的好用。

### 阶段 4：分模块推广

按 P0、P1、P2 分批推进。每批先完成模块培训页，再补页面字典，避免出现大量有字段表却没有业务上下文的页面。

### 阶段 5：使用验证

邀请新人、业务人员、测试和实施人员分别试用文档，记录无法理解、查不到或容易误解的内容，再迭代文档。

## 11. 多人协作和任务拆分

任务按四级拆分：

- Epic：模块或平台能力。
- 业务场景：例如采购收货、发料、盘点。
- 页面：申请、任务、记录、查询或配置页面。
- 证据任务：前端、后端、数据库、运行态和业务确认。

单个页面任务必须包含：

- 页面类型和使用者。
- 需要回答的业务问题。
- 输入资料和证据来源。
- 需要补充的章节。
- 当前实现与待确认项。
- 负责人、评审人和完成标准。

状态统一为：未开始、资料收集中、分析中、待业务确认、编写中、待评审、已完成、阻塞。

## 12. 资料需求和放置方式

建议将分析资料放在文档仓库同级目录，避免源代码、运行数据和凭据被误提交到产品文档仓库：

具体资料要求、实际路径登记和信息填写，请使用[资料准备清单与填写模板](资料准备清单与填写模板.md)。

```text
D:\AgentWorkSpace\
├─ MomProductDocument\
└─ MomProductSources\
   ├─ 00-版本基线\
   ├─ 01-菜单与路由\
   ├─ 02-前端源码\
   ├─ 03-后端源码\
   ├─ 04-数据库\
   ├─ 05-API与系统集成\
   ├─ 06-导入导出与打印模板\
   ├─ 07-运行态证据\
   └─ 08-业务补充资料\
```

### 12.1 启动所需资料

1. 完整菜单清单，优先使用 JSON、CSV 或 Excel。
2. 当前系统版本、部署环境和基线日期。
3. 前端源码、分支和 Commit ID。
4. 后端源码、分支和 Commit ID。
5. 数据库 DDL、迁移脚本或实体定义。

菜单清单最好包含：

`module、menu_id、parent_id、menu_name、route_path、component_path、page_code、permission_code、sort、visible、enabled、version`。

### 12.2 提高文档实用性的资料

- 当前导入模板、打印模板和标签模板。
- 脱敏后的典型业务单据和完整业务样例。
- 不同岗位的测试账号或权限矩阵。
- 已知问题、培训问题和客户常见咨询。
- 第三方接口映射、脱敏报文和失败样例。
- 业务人员当前使用的线下说明、Excel 或操作手册。

所有资料必须注明版本并完成脱敏，不保存密码、Token、私钥和生产连接串。

## 13. 资料与结论的对应关系

| 需要确认的内容 | 主要依据 |
|----------------|----------|
| 字段技术名、类型、长度和精度 | DDL/实际表结构、后端 DO/Entity、DTO |
| 中文展示名称与字段语义 | 前端标签、产品确认、业务资料 |
| 新增和编辑限制 | 后端校验、领域服务、前端交互 |
| 选择器数据源和筛选 | 前端请求、后端查询、数据权限 |
| 导入规则 | 模板、导入代码、错误样例 |
| 列表字段和顺序 | 前端列配置、运行页面 |
| 详情分组和跳转 | 前端组件、路由、产品设计 |
| 状态和动作 | 后端逻辑、工作流、运行数据 |
| 库存和单据挂接 | 事务服务、事件、表结构、运行数据 |
| 第三方接口 | 接口代码、映射配置、报文、日志 |
| 目标设计 | 产品和业务负责人评审结论 |

证据冲突时不自行猜测，登记为“当前待确认”或“待决策”。

## 14. 验收标准

文档完成不以“章节全部填满”为标准，而以实际可用性验收。

### 14.1 培训可用性

- 新人能够根据模块概述说清楚业务目的、角色、前置数据和主流程。
- 培训人员能够使用一个贯穿示例完成讲解。
- 常见操作和异常能够在文档中找到处理入口。

### 14.2 业务字典可用性

- 能够在三分钟内查到一个字段的含义、来源和维护限制。
- 技术字段名均有证据来源，不存在由中文标签自行翻译出的字段名。
- 能够查到一个单据的来源、状态、允许动作和后续影响。
- 能够查到选择器、导入、列表、详情和权限规则。
- 共享术语在不同页面中的含义一致。

### 14.3 产品改进可用性

- 所有未经确认的内容都有明确标识。
- 所有重要产品差距都有编号、影响、优先级和状态。
- 产品负责人可以直接从差距总账形成后续需求输入。

### 14.4 文档质量

- 有效菜单页面全部进入页面总账，或明确说明不纳入原因。
- 导航、页面路径和 `.linkmap.json` 保持一致。
- 每批变更通过 `mkdocs build`。
- 不新增失效链接，不直接修改 `site/` 构建产物。

## 15. 明确不做的事情

- 不把前后端源码逐段翻译成文档。
- 不为追求完整而给每个字段重复编写低价值说明。
- 不把数据库表结构直接当成培训材料。
- 不为每个页面堆叠大量截图。
- 不在具体业务页重复解释共享模型。
- 不在证据不足时编造精确规则。
- 不把产品差距和目标设想伪装成当前已实现能力。

## 16. 主要风险及控制方式

| 风险 | 控制方式 |
|------|----------|
| 内容很全但业务人员看不懂 | 先场景和示例，后字段和技术信息 |
| 文档成为字段堆砌 | 模块培训页和业务链说明先行 |
| 当前实现与产品设想混淆 | 使用统一结论标识和证据来源 |
| 页面数量过大导致失控 | 按 P0/P1/P2 分批并先完成试点 |
| 多人编写导致术语不一致 | 建立共享模型和业务术语字典 |
| 文档完成后无人使用 | 增加新人、业务、测试和实施试用验收 |
| 源码与部署版本不一致 | 每页记录适用版本和证据基线 |
