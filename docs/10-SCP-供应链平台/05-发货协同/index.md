# 发货协同

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；供应商发货、采购跟单（顺带）。

## 业务目的与适用范围

发货申请提交了，仓库却说没收到单——先别急着重发，多半是 **SCP→WMS 推送接口没成功**。发货协同覆盖供应商发货申请与发货记录（含 ASN 号、到仓/月台、承运与车牌、计划到货时间等）：申请提交并形成记录后，通过接口推送发货主/明细（及批次属性等），供仓库安排收货；记录可带「是否已生成采购收货申请」标志与申请号。

读完本页，应能判断：

1. 「仓库无单」是申请/记录本身有问题，还是推送未成功；
2. 数量超发卡在订单/计划未交量，还是选错了来源；
3. 仓内收货、检验、成品客户发货各归哪一侧，避免混用「发货」一词。

旧稿「提交即生成 ASN 并保证 WMS 自动完成」须以接口成功为准；失败应查接口信息，不能假定库存已变。操作步骤见[发货协同-维护与查询参考](发货协同-维护与查询参考.md)。

## 使用前准备

| 需要确认什么 | 为什么重要 |
| --- | --- |
| 已发布/可发的订单或计划 | 申请挂订单号 / 计划号；无可发来源则建不了合法申请。 |
| 物料包装、批次属性 | 扫码与收货核对；漏维会影响现场核对。 |
| 到仓、月台、时间窗、承运信息 | 决定到货安排；到错仓仓库无法按预期收。 |
| 接口通道可用 | 推送失败则 WMS 侧无 ASN，仓库表现为「无单」。 |

## 一笔发货如何完成

```mermaid
flowchart LR
    A["订单/计划可发量"] --> B["发货申请"]
    B --> C["提交/处理"]
    C --> D["发货记录 + ASN"]
    D --> E["SCP→WMS 推送"]
    E --> F["WMS 收货执行"]
    F --> G["跟踪收货结果"]
```

申请表达「要对哪一批可发量发起发运」；提交/处理后形成发货记录与 ASN，再经接口推给 WMS。发货完成时会回写采购计划已发数量等（服务内已有同步线索）；撤销发货存在重新推送 WMS 的路径。仓内任务、库存事务与 PDA 收货不在本页展开，见采购收货。

!!! example "📝 示例数据占位"
    供应商按已发布订单 PO-2001 发运物料 A 共 50 件。展示申请号、ASN、推送结果、是否已建收货申请标志，以及订单/计划已发数量变化。

!!! example "写实示例：给定 → 期望（含推送分支）"
    **给定：** 采购订单 PO-2001 已发布，物料 A 剩余可发 50；供应商新建发货申请到仓 W1、计划到货明日，明细 50；提交后形成发货记录与 ASN。

    **期望（推送成功）：**

    1. 发货记录可查到 ASN；接口信息中 SCP-WMS 发货主/明细登记成功。
    2. WMS 侧可见对应到货/收货来源（按环境；是否同实例见 `GAP-017`）。
    3. 订单/计划已发数量增加；超可发量的申请会被拒绝或提示，不能无提示地超发。
    4. 若环境已启用由发货记录建收货申请，记录上「是否已建收货申请」与申请号有值；未启用则标志可为空，不以标志为空反证推送失败。

    **期望（推送失败）：**

    1. SCP 侧仍可能已有发货记录/ASN，但仓库查不到单——先查接口信息是否成功，不要假定「重提一次申请」就能修好。
    2. 按环境提供的重推 / 撤销后重发路径处理；勿直接改 WMS 库存或手工在仓内伪造 ASN。
    3. 修数或重推成功后，再核对 WMS 可见性与已发数量回写。

## 主对象

| 对象 | 业务含义 | 使用者最关心什么 |
| --- | --- | --- |
| 发货申请头/行 | 申请号、关联计划/订单、到仓月台、时间窗、承运、明细数量与包装。 | 来源是否可发、数量是否超未交、到仓是否正确。 |
| 发货记录头/行 | ASN、执行时间、申请号回挂、是否已建收货申请等。 | ASN 是否形成、推送是否成功、下游是否已建收货申请。 |
| 发货检验明细 | 协同侧检验线索（与 QMS 权威边界待确认）。 | 是否仅线索、是否自动建 IQC（❓ `GAP-017`）。 |
| 供应商包装 | 发货包装辅助。 | 标签/包装是否可核对。 |

申请状态复用通用请求状态枚举（NEW、REVIEWING、AGREED、…、COMPLETED 等）。

## 与 WMS / QMS / 销售发货边界

「发货」一词在采购协同和销售出库里各指一件事，下表避免混用：

| 协同方 | 本页负责 | 不在本页展开 |
| --- | --- | --- |
| WMS 采购收货 | 推送 ASN/发货记录；可触发收货申请标志 | 库内任务、库存事务、PDA 收货 |
| WMS 成品发货 | — | 客户发货申请/任务/记录（另一菜单域） |
| QMS 来料检验 | 可选检验明细线索 | IQC 申请任务记录权威 |
| 采购跟踪 | 发货后跟踪未收/已收 | 退货/短缺细则 |

本页没有独立策略页；可发范围主要受上游订单/计划状态与未交量约束，下游可见性受 SCP→WMS 接口成败约束。变更来源单据状态或接口通道前，应在测试环境走通「申请 → 记录 → 推送 → 仓库可见」并留回退/重推路径。

## 关键判断

| 判断点 | 应先确认什么 | 判断后的影响 |
| --- | --- | --- |
| 数量是否超发 | 订单/计划未交量、行是否仍可发 | 申请校验失败或超发拒收 |
| 仓库为何无单 | 发货记录是否形成 ASN；接口状态是否成功 | 决定修申请/重推，还是查仓内菜单权限 |
| 是否重复建收货申请 | 「是否已建收货申请」标志与申请号 | 避免重复申请；标志空不等于推送失败 |
| 推送失败如何处理 | 接口信息、环境重推/撤销重发入口 | 勿直接改 WMS 库存；先协同侧修数再推 |

### 建议验证点

- 基于已发布订单或已接受计划创建发货申请，提交后形成记录与 ASN。
- 推送成功：接口登记成功，且 WMS 侧可见对应到货/收货来源（按环境）。
- 推送失败：SCP 有记录但仓库无单；按重推/撤销重发路径恢复后，仓库可见性应追上。
- 超可发量申请被拒绝或提示；成功发货后订单/计划已发数量增加。
- 「是否已建收货申请」在启用/未启用两种环境下的表现（启用条件 ❓ `GAP-017`）。
- 答疑「仓库无单」时：先查 ASN 是否形成，再查接口成败，最后才查仓内权限/菜单。

## 限制与待确认

- 发货申请与记录是否必须一对一、部分发货多次：以环境配置与测试为准（`GAP-017`）。
- 发货检验明细与 QMS 来料检验是否自动建单：**未证实**（`GAP-017`）。
- 由发货记录自动创建 WMS 收货申请的启用条件：待联调（`GAP-017`）。
- 可发订单/计划精确状态集与选择器投影待测（`FSEM-006`）。
