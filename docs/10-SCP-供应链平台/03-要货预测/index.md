# 要货预测

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；计划员、供应商门户（顺带）。操作细节见[要货预测-维护与查询参考](要货预测-维护与查询参考.md)。

## 业务目的与适用范围

供应商说「按旧版本备货了」——多半是没盯紧当前活动版本。要货预测处于协同链前端：**基础数据就绪之后、要货计划之前**，用于中长期需求发布与供应商对齐。计划方维护预测主单与明细（版本、窗口、计划量等），供应商侧有独立查看/确认页面；预测可为后续采购计划提供输入，接口类型枚举含预测主/明细，可登记对外同步。

读完本页，应能判断：

1. 该催「发新版本」还是「等供应商读/确认活动版本」；
2. 预测本身**不是**可发货计划，也**不是**采购订单；
3. 自动汇总进计划是否启用——未证实前勿当默认闭环（`GAP-017`）。

## 一笔预测如何协同

```mermaid
flowchart LR
    A["计划员维护预测主/明细"] --> B["发布版本"]
    B --> C["供应商查看/确认"]
    C --> D["汇总进入采购计划（按业务配置）"]
```

计划方发活动版本 → 绑定供应商的门户用户可见 → 已读/确认（按环境）→ 按策略手工或自动汇总进采购计划。汇总规则未证实前，培训以「已发布可被引用」表述，勿写成「发布即自动成计划」。

!!! example "写实示例：给定 → 期望"
    **给定：** 预测 FC-2026-07，供应商 V-01，活动版本 V3，窗口为本月～下月；明细物料 A 计划量 1000；发布后供应商用户已读但未改量。
    **期望：**

    1. 发布前供应商侧通常看不到该版本；发布后可见活动版本 V3，勿用旧版 V2 备货。
    2. 「已读」≠「已确认可改量」——可确认字段集以环境为准（`GAP-017`）。
    3. 若策略未开自动汇总，需按配置手工纳入[要货计划](../04-要货计划/index.md)；开了则核对计划侧是否出现对应行。
    4. 预测行上的计划量不是发货可发上限；可发量以计划接受后的行量 / 订单未交为准。

## 建议验证点

- 计划方发布后，绑定该供应商的用户可见对应活动版本；未发布不可见。
- 切换活动版本后，协同与催办以新版本为准，旧版勿当现行需求。
- 已读标记可更新（若启用）；勿把已读当成已确认改量。
- 按环境验证「自动汇总 / 手工纳入计划」两种路径（自动规则 ❓ `GAP-017`）。
- 从预测可追溯到后续采购计划（按配置）；预测量本身不能直接当发货上限。

## 限制与待确认

- 供应商确认数量字段与驳回路径的完整按钮集：以测试环境为准（`GAP-017`）。
- 预测→计划的自动/手工规则：依赖策略配置，待联调（`GAP-017`）。
- 与 MES 是否消费本预测：**未证实**。
