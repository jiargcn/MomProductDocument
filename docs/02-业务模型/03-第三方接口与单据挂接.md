# 第三方接口与单据挂接

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 目标：说明各业务单据如何与 ERP、MES、供应商平台、设备或其他第三方系统关联，并可追溯请求、结果与失败处理。
> 平台可靠性（调用记录、异步重试、日志分层）见[数据交换与集成可靠性](../03-基础设施/09-数据交换与集成可靠性.md)。

## 本页解决什么问题

集成问题常被拆成两处却对不上：业务页只写「有接口」，平台页只写「能重试」。本页界定 **业务挂接**——哪个单据、何时触发、字段/状态如何映射、失败后业务侧如何补偿——并与平台可靠性分工。

读完应能判断：当前问题该查业务页的触发与映射，还是查基础设施页的调用轨迹与重试；未核验映射不得写成现状。

## 如何使用本页

| 你的目的 | 建议做法 |
| --- | --- |
| 理解实时 vs 延时交换 | 读「两类数据交换」表 |
| 写业务页时引用 | 写清触发时点、关联键、失败补偿；平台重试细节链基础设施页 |
| 排障时对照 | 先定位内部单号 ↔ 外部单号；再查调用记录/异步失败（平台页） |
| 验收集成闭环 | 对照「待填写任务」完成标准；未证实项保留 `❓` |

## 参考设计：两类数据交换

| 模式 | 参考适用场景 | 设计要点 |
| --- | --- | --- |
| 实时交换 | 需要在同一业务动作中立即取得外部结果的场景 | 直接调用外部 API；超时、失败和事务边界必须明确。 |
| 延时交换 | 可异步处理、需要削峰或可靠补偿的场景 | 业务单据生成时创建交换数据，经消息与独立交换存储转发；需有状态、重试和对账。 |

## 与平台能力的分工

| 层级 | 写什么 | 页面 |
| --- | --- | --- |
| 业务挂接 | 哪个单据、何时触发、字段/状态映射、业务补偿 | 本页 + 各业务页 |
| 平台可靠性 | 调用轨迹、异步成功/失败、人工重试、日志联查 | [数据交换与集成可靠性](../03-基础设施/09-数据交换与集成可靠性.md) |

## 自检或验证点

1. **能定位：** 从内部单据能否找到外部单号/关联键，以及最近一次交换结果？
2. **失败可恢复：** 超时或失败后，业务补偿与平台重试各自负责什么？
3. **未证实不伪装：** 采购收货等外部来源分支的报文与回写，是否仍随业务页续作，而非写成全站已闭合映射？
