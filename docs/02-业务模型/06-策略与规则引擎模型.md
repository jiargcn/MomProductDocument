# 策略与规则引擎模型

> 状态：占位页已建立。策略框架为旧版规划书的稳定设计理念；当前策略实体与命中逻辑待核验。

## 本页解决什么问题

收货时段、质检方式、上架库位、拣选、补货、盘点、容量和接口处理等，往往不是写死在单个业务页里，而是由 **策略 / 规则** 在适用范围内命中后给出允许、推荐或控制结果。本页统一说明策略如何定义、命中和治理；具体「本业务用了哪条」写在业务页。

读完应能判断：业务行为异常时该查策略配置还是查单据/任务配置；写业务页时应写「引用哪项策略」，并把未证实的命中逻辑标 `❓`。与单据侧规则挂接的关系见[单据类型、业务类型与单据配置](05-单据类型、业务类型与单据配置.md)。

## 如何使用本页

| 你的目的 | 建议做法 |
| --- | --- |
| 理解策略由什么构成 | 读「参考设计」四条 |
| 写业务页时引用 | 业务页写「使用哪项策略 / 规则」；定义与命中通则链本页 |
| 排障时对照 | 先确认策略域与适用范围，再查优先级/默认规则是否冲突（命中细节待核验） |
| 验收配置变更 | 变更是否具备权限、审计、生效边界——当前以待确认项为准 |

## 参考设计

1. 一条策略由适用范围、条件、优先级和输出结果构成。
2. 条件通常包含参数、比较符和目标值；输出决定允许、推荐或控制的动作。
3. 同一策略域应有确定的优先级与默认规则，避免重复优先级导致结果不确定。
4. 策略可能影响业务执行，故配置变更应具备权限、审计和生效边界。

## 自检或验证点

1. 业务页是否只写「有策略」而未点名策略域 / 配置入口？（若未取证，应标 `❓`）
2. 是否把旧规划书的策略分类直接写成当前已落地清单？
3. 配置变更后生效方式、命中日志是否仍在待确认，而非假装已有运维手册？
