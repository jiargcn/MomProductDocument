# 库存数据挂接模型

> 状态：WMS 采购收货—库存链路已完成首轮取证；销售出库主链已部分证实；其余业务待核验。
> 目标：统一说明预计入、预计出、库存事务、库存余额如何由业务单据创建、消耗、追溯和冲销。

## 本页解决什么问题

「任务还在等收货」「收货完了账为什么变了」「现在还能不能发」——这三类问题分别落在预计、事务、余额上，却常被混成一张库存表。本页说明这三类对象如何挂接申请/任务/记录，以及过账时序与追溯方向。

读完应能判断：某业务应在哪一时点看预期、哪一时点看事务/余额；排障时该「余额 → 事务 → 记录 → 任务」还是反过来；业务页不得把「余额查询」写成库存变化来源。

粒度与选择器口径见[库存管理精度与唯一粒度](08-库存管理精度与唯一粒度.md)；查询入口见[库存管理](../05-WMS-库房管理/09-库存管理/index.md)。

## 当前已知事实

采购收货任务创建预计入，采购收货记录创建库存事务，库存事务服务更新库存余额。库存余额业务唯一粒度已确定为批次、托盘、包装、物料、库存状态、库位。

销售出库主链已证实：处理申请生成任务时建立预计出，发货记录形成出库事务并更新余额，同时清理对应预计出。发料等其它出库业务对预计出的创建时点仍待逐项核验。

### 采购收货过账时序（样板）

```mermaid
sequenceDiagram
    participant App as 收货申请
    participant Job as 收货任务
    participant Exp as 预计入
    participant Rec as 收货记录
    participant Txn as 库存事务
    participant Bal as 库存余额

    App->>Job: 处理 / 自动执行生成任务
    Job->>Exp: 创建预计入（待完成到货预期）
    Note over Job,Exp: 挂接任务号；尚非实际库存
    Job->>Rec: 现场执行完成
    Rec->>Txn: 按实收生成库存事务
    Txn->>Bal: 按余额业务键更新当前数量
    Note over Exp: 任务关闭/完成/撤销时清理或释放预期（细节见业务页）
    Rec-->>App: 可继续上架 / 检验 / 退货 / 外部回写
```

追溯顺序与过账相反：余额 → 最近事务 → 业务记录 → 任务 / 来源申请。不要把预计入当作已入账数量，也不要用手工改余额代替源业务。

## 参考设计：数据责任

| 数据对象 | 参考责任 | 与单据的挂接方式 |
| --- | --- | --- |
| 预计入/预计出 | 表示已计划但未实际发生的供需占用 | 通常由待执行任务创建，在完成、取消或变更时相应消耗或释放。 |
| 库存事务 | 记录一次可追溯的库存变化 | 应关联业务记录、事务类型、数量方向及来源单据。 |
| 库存余额 | 表示当前可用/冻结等库存状态下的汇总事实 | 只能由库存事务等受控业务动作改变，不能以查询或人工汇总代替。 |
| 移动日志 | 记录物理移动或状态变化轨迹 | 用于追溯，是否独立实现待当前基线核验。 |

## 自检或验证点

1. **分层：** 「还在等什么」查预计；「为什么变了」查事务；「现在剩多少」查余额。
2. **挂接：** 预计能否按任务号查到？事务能否按记录号回查业务？
3. **禁止误写：** 业务页是否避免把「库存余额查询」写成变化来源？
4. **未证实路径：** 发料等其它出库的预计出时点，是否仍标待核验而非写成采购收货对称结论？

## 输出规则

页面不得把「库存余额查询」误写为库存变化来源；库存变化必须经事务追溯。数据库约束不足、并发控制和无法确认的粒度须进入产品差距总账。
