# 主数据治理模型

> 状态：占位页已建立。治理目标来自旧版规划书；当前主数据域和变更流程待 DBC、System 源码核验。

## 本页解决什么问题

单据、库存、质量、设备与接口都引用同一批主数据。若各表各自约定「能不能删、能不能停用、外部谁说了算」，培训和排障会对不上。本页给出主数据 **分类、归属、变更、启停与归档** 的共同依据；各主数据页写本域细则。

读完应能判断：写业务页约束时该引用哪些治理原则；遇到「删不掉 / 选不到 / 外部同步冲突」时先查主数据启停与引用关系，而不是只查业务单状态。选择器侧通例见[通用选择器过滤惯例](12-通用选择器过滤惯例.md)。

## 如何使用本页

| 你的目的 | 建议做法 |
| --- | --- |
| 理解治理原则 | 读「参考设计」表 |
| 写业务页时引用 | 新增/编辑/导入约束对齐本页原则；本域编码与启停细则写在 DBC 业务页 |
| 排障时对照 | 选不到 → 先查可用/停用与权限；删改失败 → 先查是否被单据/库存/接口引用 |
| 验收变更流程 | 对照待确认项；未核验前不要写成统一审批流已落地 |

## 参考设计

| 治理主题 | 参考要求 |
| --- | --- |
| 维护与引用 | 主数据变更须识别被哪些单据、库存或外部接口引用。 |
| 启停与归档 | 优先采用受控停用或归档，避免直接删除影响历史事实。 |
| 外部集成 | 明确外部来源、同步方向、冲突解决和主数据责任方。 |
| 查询与审计 | 支持关联数据查询、变更追溯和必要的审批。 |

## 自检或验证点

1. 业务页是否把「直接删除主数据」写成常规操作，而未提引用与启停？
2. 外部同步冲突时，是否写清责任方与冲突解决（未证实则 `❓`）？
3. 选择器「选不到」是否同时考虑停用与权限裁剪（见选择器惯例页）？
