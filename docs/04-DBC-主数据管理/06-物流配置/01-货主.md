# 货主

> 适用基线：测试环境 / `dev` 分支 / 2026-07-15。

## 这项主数据解决什么问题

货主用于表达库存或业务对象的归属主体。当同一仓储网络需要区分不同委托方、法人主体或货权主体时，货主让收发货、库存查询和结算相关业务能够使用同一归属口径。

货主不是供应商或客户的替代名称：供应商描述供货关系，客户描述交付关系，货主描述货物/业务归属。三者可能是同一组织，也可能完全不同，不能仅凭名称直接复用。

## 什么时候需要维护

| 业务事件 | 应做什么 | 维护前要确认 |
| --- | --- | --- |
| 新增委托库存或需区分货权 | 新建货主并明确归属范围。 | 归属主体的正式名称、简称、类型和生效日期。 |
| 联系、结算或税务信息变化 | 更新相应信息并记录变更原因。 | 是否影响对账、接口或外部单据。 |
| 主体不再适用 | 停用并保留历史记录。 | 是否存在库存、在途业务或历史查询引用。 |
| 一次维护多个主体 | 使用导入模板并逐行核对结果。 | 代码、简称、类型和可用状态已准备。 |

## 它与物流业务的关系

```mermaid
flowchart LR
    A["货主"] --> B["库存或业务归属"]
    B --> C["收货、发货与库存查询"]
    C --> D["对账、追溯与分析"]
```

当前资料只证实货主主数据可维护，尚未逐一确认哪些收发货、库存或结算页面已经引用它。因此，图中是应验证的业务关系，不代表全部业务均已自动挂接。

## 关键字段业务角色

| 字段/配置点 | 行为模式 | 在系统中的作用 | 关键行为要点 | 维护时要警惕什么 |
| --- | --- | --- | --- | --- |
| 货主代码 | P6 | 归属主体识别号 | 新增必填；服务层按代码查重 | 已被库存/单据引用后勿改码 |
| 简称 | P6 | 日常识别 | 新增必填 | 与供应商/客户简称区分使用 |
| 类型 / 币种 / 税率 | P1 | 属性与结算口径 | 选择范围与合法性 ❓ | 不确定时先确认物流/财务口径 |
| 是否可用 / 有效期 | P1 / P10 | 生命周期 | 可用必填 | 停用前查库存与在途业务 |

## 维护与查询说明

| 维护事项 | 当前可确认做法 | 操作提示 |
| --- | --- | --- |
| 新增 | 代码、简称和可用状态为必填；可维护名称、地址、联系、银行、币种、税率、类型、生效/失效时间和备注。服务层会查重代码。 | 先确认“归属主体”而非采购/销售伙伴的业务含义。 |
| 编辑 | 联系、地址、银行、币种、税率和有效期可按业务变更维护；服务仍可写回代码等关键字段。 | 已被业务引用后，代码、简称和类型的修改应先做影响评估；日常按“代码不改”执行。 |
| 停用 | 通过可用状态控制后续使用。 | 不把停用当删除；先确认库存、在途单据与历史查询。 |
| 导入 | 模板校验代码、简称、可用；按代码匹配既有行后按模式写入。导入路径对代码唯一性的校验弱于普通新增。 | 类型、币种使用前应确认当前可选口径；导入后抽查归属信息。 |
| 删除 | 仅校验存在后删除；未见下游引用拦截。 | 有库存或在途引用时优先停用。 |
| 查询 | 列表优先显示代码、简称、名称、类型、币种、可用状态和有效期。 | 常用筛选为代码/名称、类型、可用状态和有效期。 |

建议详情页分为“基本识别、联系与地址、结算信息、类型与有效期、变更记录”五组；现有页面的实际分组和快速跳转需补充截图验证。

## 常见问题与联查

| 现象 | 优先检查 | 后续联查 |
| --- | --- | --- |
| 不确定应建货主还是供应商/客户 | 该主体是货物归属，还是供货/交付对象。 | 供应商、客户与目标收发货业务。 |
| 业务中无法选择货主 | 可用状态、有效期和主体类型。 | 目标业务页面的选择条件与权限。 |
| 历史数据归属显示不一致 | 货主变更时间和已引用业务。 | 收发货、库存或对账记录。 |

## 当前边界与待确认事项

- 代码唯一性目前主要依赖服务层查重，数据库层业务唯一约束与并发表现仍待验证。
- 导入路径对重码校验弱于普通新增；类型/币种选择范围和已引用后的编辑限制尚未完成测试验证。
- 当前尚未确认货主在各库存、收发货、结算和第三方接口中的实际引用范围。
- 建议总账保持/登记：货主引用地图、导入与新增校验对齐、引用后改码/停用保护（可沿用内部 `DBC-OWNER-*` 主题）。
