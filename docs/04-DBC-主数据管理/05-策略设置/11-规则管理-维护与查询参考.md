# 规则管理-维护与查询参考

> 本页是[规则管理](06-规则管理.md)的操作参考。规则变更必须同时提供业务口径、测试样例、发布计划和回退方式。

## 新增与编辑约束

| 信息组 | 应维护的信息 | 维护限制 |
| --- | --- | --- |
| 适用范围 | 业务类型、规则分类。 | 先确认规则服务的业务，不得以“试试效果”为目的在线上随意配置。 |
| 版本与状态 | 版本、状态。 | 版本应能区分替代关系；状态切换前需完成回归验证。 |
| 判断内容 | 条件、规则配置/节点编排内容。 | 条件必须可由业务样例验证；复杂配置应由具备权限的人员维护。 |
| 补充说明 | 备注。 | 写明变更原因、适用边界和回退依据。 |

当前尚未确认哪些字段在发布后锁定、是否支持版本复制及规则冲突时的优先级；这些事项不得凭页面名称推定。

## 导入规则

| 模板应包含 | 不应作为普通批量维护项 | 导入前校验 |
| --- | --- | --- |
| 业务类型、规则分类、版本、状态、条件，以及必要的规则配置与备注。 | 实际业务处理结果、运行日志、未经验证的复杂配置文本。 | 业务类型与分类合法；版本命名一致；条件可解析并有命中/不命中样例；状态符合发布流程。 |

复杂规则应先在少量样例中试导，导入后逐条验证；如果导入结果无法区分“保存成功”和“业务已生效”，须补充验证记录。

## 验收清单

- [ ] 规则在正确的业务类型和分类下可被查询。
- [ ] 至少一组样例命中、另一组样例不命中，结果符合业务预期。
- [ ] 发布/停用后的业务结果与状态一致。
- [ ] 新版本切换后，旧版本的历史追溯和回退方式明确。
- [ ] 导入失败时，能识别失败行和失败原因。
