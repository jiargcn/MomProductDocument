# 规则管理

> 适用基线：测试环境 / `dev` 分支 / 2026-07-15。
> 具体新增、编辑、导入和查询操作见[规则管理-维护与查询参考](11-规则管理-维护与查询参考.md)。
> 共享分流语义见[单据类型、业务类型与单据配置](../../02-业务模型/05-单据类型、业务类型与单据配置.md)。

## 这项配置解决什么问题

规则管理用于把特定业务类型下的判断条件、规则分类、版本、状态和配置内容集中维护。它服务于需要按条件决定处理方式的业务，而不是取代业务人员对异常单据的判断。

规则一旦被业务流程引用，修改可能改变后续单据、任务或记录的处理结果。因此，规则应视为“可发布的业务决策”，而不是普通文本配置。

## 关键字段业务角色

| 字段/配置点 | 行为模式 | 在系统中的作用 | 关键行为要点 | 维护时要警惕什么 |
| --- | --- | --- | --- | --- |
| 业务类型 | P12 / P2 | 规则所属场景 | 必填；决定规则面向哪类业务 | 挂错业务类型 |
| 分类 / 条件 | P12 / P1 | 触发判断 | 必填；须可验收样例 | 条件不可测 |
| 版本 / 状态 | P9 / P12 | 发布生命周期 | 后端有版本化模型；发布锁定多为注释代码（`GAP-063`） | 勿假定已发布不可改已强制 |
| 规则配置 / 节点 | P12 | 执行内容 | **Web 字段与后端模型可能不一致**（`GAP-063`） | 按错误字段培训 |
| 策略代码 / 优先级等 | — | 残留/Web 口径 | 可能与 DDL 不一致 | 历史口径勿当当前 |

完整语义见[维护与查询参考](11-规则管理-维护与查询参考.md)。

## 什么时候需要维护

| 业务事件 | 应做什么 | 维护前要确认 |
| --- | --- | --- |
| 新场景需要统一判断 | 新建规则并明确所属业务类型、分类、条件和预期结果。 | 业务口径、适用范围和验收样例已评审。 |
| 判断逻辑发生变化 | 建立新版本或受控变更，并做回归验证。 | 已使用规则的历史追溯和回退方式。 |
| 规则暂不应执行 | 调整规则状态，保留停用原因。 | 在途业务和异常处理是否仍依赖该规则。 |
| 规则数量较多 | 采用模板导入前先确认导入端点是否可用（`GAP-063`）。 | 条件表达已在测试环境验证。 |

## 规则如何进入业务

```mermaid
flowchart LR
    A["业务类型"] --> B["规则分类与版本"]
    B --> C["条件与规则配置"]
    C --> D["业务处理判断"]
    D --> E["申请、任务或记录结果"]
```

图中只说明规则影响业务的通用结构。不同分类的实际处理动作、触发时点和优先级尚未全部确认，不能据此推定某条规则已经在所有业务中生效。

## 维护时最重要的判断

| 需要判断什么 | 业务含义 | 建议做法 |
| --- | --- | --- |
| 所属业务类型是否正确 | 决定规则面向哪类业务。 | 不在不相干的业务类型中复用相同规则。 |
| 分类与版本是否可区分 | 使同类规则可维护、可回溯。 | 版本变更写明生效原因和替代关系。 |
| 条件是否有可验收样例 | 条件决定何时触发。 | 至少准备命中和不命中两组脱敏样例。 |
| 状态是否应切换 | 状态会影响规则是否参与业务判断。 | 与业务负责人共同验证后再发布或停用；注意发布限制可能未强制（`GAP-063`）。 |

## 查询与联查

| 想回答的问题 | 建议先查什么 | 再联查什么 |
| --- | --- | --- |
| 某业务为什么得到这个处理结果 | 业务类型、规则分类、版本、状态和条件。 | 对应申请、任务或记录的实际结果。 |
| 哪条规则正在使用 | 可用规则的状态、版本和适用范围。 | 业务类型、单据设置及测试样例。 |
| 规则变更影响哪些业务 | 变更时间、条件和配置差异。 | 变更前后同一业务样例的处理结果。 |

## 当前边界与待确认事项

- 前后端/导入多套字段口径（`GAP-063`）；Web 导入按钮可能无对应端点。
- 不同规则分类的触发时点、优先级、冲突处理和回退行为尚未完成端到端验证。
- 已发布不可更新、非草稿不可删除等限制多为注释，不得写成已强制门禁。
