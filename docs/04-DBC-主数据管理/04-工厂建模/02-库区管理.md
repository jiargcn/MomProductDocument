# 库区管理

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；仓储主数据与上架/拣选人员（顺带）。
> 具体新增、编辑、启停、导入和查询操作见[库区管理-维护与查询参考](15-库区管理-维护与查询参考.md)。

## 业务目的与适用范围

库区是仓库内部按用途、状态或作业方式划分的区域，例如收货区、存储区、拣选区、隔离区。它帮助业务在「同一仓库」内继续区分物料应在哪里处理、哪些库存可用以及作业应从哪个区域开始。

读完本页，应能判断：库区必须挂在正确仓库下、类型名称**不等于**自动隔离策略，以及收货/上架**选不到库区**时是仓未选、区挂错仓，还是区已停用。实际库存可用性仍需结合库存状态、库位和质量结论判断。

## 何时需要维护

仓库启用新功能区、收发流程调整、需要隔离/待检管理，或物料库区配置与实际作业地点不一致时，应维护库区资料。

## 使用前准备

| 需要确认什么 | 为什么重要 |
| --- | --- |
| 所属仓库已存在且可用 | 库区必须挂在仓库下；错挂导致整区作业错位。 |
| 库区编码、名称与类型 | 区域识别与用途口径；类型不自动强制库存状态。 |
| 是否可用 | 停用前查下属库位、库存与任务。 |
| 下属库位计划 | 有区无位时，落到库位的收货/上架仍无法闭环。 |

## 库区如何服务作业

```mermaid

    A["仓库"] --> B["按用途划分库区"]

```

库区体现业务用途，不等于某一条库存记录。

!!! example "写实示例：给定配置 → 期望行为"
    **给定：** 仓库 `WH-RM` 可用；新建库区 `RCV`（收货区）挂在该仓，是否可用 = 是；其下已有库位 `RCV-01`。
    **期望：**

    1. 业务页先选仓库 `WH-RM` 后，库区选择器应只出示该仓下库区（含 `RCV`）；未选仓库时通常选不到或列表为空（通例）。
    2. 变更业务页已选仓库时，已选库区应按通例**清空**后再按新仓过滤，禁止跨仓残留。
    3. 将 `RCV` 挂错到其它仓库，或库区停用后，收货/上架在正确仓下应选不到该区（过滤时点 ❓）。
    4. 库区类型名为「隔离」时，**不能假定**系统已自动禁止可用库存进入；隔离仍依赖库存状态、质量结论与任务规则。

## 关键字段业务角色

完整语义、选择器与联动见[维护与查询参考](15-库区管理-维护与查询参考.md)。「先仓后区」级联通例见[库位与仓储级联惯例](../../02-业务模型/13-库位与仓储级联惯例.md)；「仅可选已存在/可用仓库」通例见[通用选择器过滤惯例](../../02-业务模型/12-通用选择器过滤惯例.md)。本页差异：编辑常锁仓库、类型不自动强制库存状态。

| 字段/配置点 | 在系统中的作用 | 关键行为要点（取值/范围/联动/门禁） | 维护或操作时要警惕什么 |
| --- | --- | --- | --- |
| 库区编码 | 区域内地点识别 | 新增必填；Web 编辑通常锁定代码；唯一性 ❓（`GAP-049`） | 已挂库位后勿随意改码 |
| 所属仓库 | 库区的上级地点 | **仅可选已存在仓库**；编辑通常锁定仓库；变更仓库应先迁移下级 | 错挂仓库导致整区作业错位 |
| 库区类型 | 收货/存储/拣选/隔离等用途口径 | 字典；**不能假定**类型会自动限制库存状态或上架策略 | 勿只靠类型名称推断强制隔离 |
| 管理精度 / 部门 | 管理粒度与组织归属资料 | 部门不能代替仓库归属；精度与策略挂接 ❓ | 误把部门当地点过滤条件 |
| 是否可用 / 有效期 | 控制区域是否可被引用 | 停用前查下属库位、库存与任务 | 有库存仍停用 → 作业中断 |

### 关键维护与变更摘要

| 维护点 | 业务判断 | 使用建议 |
| --- | --- | --- |
| 归属仓库 | 库区属于哪个仓库。 | 避免跨仓重复或错挂。 |
| 区域用途 | 是否用于收货、存储、拣选、隔离或其它作业。 | 用途应与实际流程和人员分工一致。 |
| 下属库位 | 是否能满足区域内的实际存放和扫码操作。 | 先建区域再按需要建库位。 |
| 启停/调整 | 是否仍有库存或在途任务。 | 调整前先迁移库存和作业。 |

### 归属与可用性如何起作用

| 维护点 | 起作用的方式 | 排障时先看 |
| --- | --- | --- |
| 所属仓库 | 业务选区前必须先有正确仓库；区从属于仓 | 区是否挂在当前所选仓下 |
| 是否可用 | 停用后原则上不应再被新任务选中；过滤时点 ❓ | 库区是否停用 |
| 库区类型 | 用途理解；不自动等于库存状态门禁 | 是否误把类型当强制策略 |

现场说「上架选不到库区」：① 是否已选仓库；② 库区是否属于该仓且可用；③ 再查任务是否限制改区/改位。

### 建议验证点

- 未选仓库时，库区选择器是否为空或不可用。
- 换仓后已选库区是否被清空并按新仓重过滤。
- 库区挂错仓时，正确仓下是否选不到该区。
- 停用库区后，收货/上架选区是否立即过滤（时点 ❓）。
- 类型为隔离/待检时，是否仍可把可用库存上到该区（期望：不能只靠类型名拦截）。

## 查询、详情与联查

可从仓库查库区，从库区查库位、库存余额和当前作业；发现上架或拣选地点不合理时，应同时检查物料库区配置、库存状态和任务规则。

## 维护与查询边界

库区必须先明确所属仓库和用途，再建立库位组、库位和作业规则。当前支持批量导入，导入时应按“仓库先于库区”的顺序进行；部门归属可作为管理信息维护，但不能代替仓库归属。

当前资料不能证明库区类型会自动限制库存状态、物料范围或上架策略。需要隔离、待检、拣选等强制控制时，必须同时验证库存状态、任务规则与现场操作，不能只依赖库区名称。

## 常见问题与处理

| 情况 | 建议处理 |
| --- | --- |
| 任务找不到目标区域 | **排障入口：** 先确认任务/单据已选仓库 → 本页核对库区归属与可用 → 再查下属库位、物料配置与任务条件。 |
| 待检/隔离库存进入普通区 | 停止后续使用，核对库存状态、质量结论和库区用途；不要只改库区名称。 |
| 调整库区影响库存 | 先安排调拨/转库并处理未完成任务。 |

## 当前限制与待确认事项

- 普通保存与导入校验路径不一致（`GAP-049`）；勿把导入规则写成页面保存的既成约束；
- 库区类型是否强制限制库存状态和物料范围需测试；
- 需补列表、详情和任务引用截图。
