# 供应商

> 适用基线：测试环境目标 / `dev` 分支 / 2026-07-15。
> 阅读对象：测试、实施、运维（主）；主数据与采购协同人员（顺带）。

## 业务目的与适用范围

供应商主数据统一识别企业的供货方。采购选源、仓库核对到货方、维护供应商物料时，都应引用同一份资料，而不是在每张单据上重复录入名称与联系信息。

读完本页，应能判断：什么情况下必须先建供应商、停用或改码会影响什么，以及**采购/收货选不到供应商**时先查主体状态还是先查物料关系。字段填写、批量导入、列表筛选见[供应商-维护与查询参考](03-供应商-维护与查询参考.md)。「谁能供什么料」见本分组的[供应商物料](02-供应商物料.md)。

## 使用前准备

维护一项供应商前，至少应由采购或主数据责任人确认：

| 需要确认什么 | 为什么重要 |
| --- | --- |
| 供应商代码与简称 | 代码是系统识别供应商的稳定标识；简称是日常业务快速辨识信息。 |
| 正式名称与联系信息 | 便于采购、到货异常和日常协同找到正确对象。 |
| 币种、类型、税率与生效区间 | 用于区分供应商属性和适用期间；具体是否在某业务中生效，仍以对应业务规则为准。 |
| 是否可用 | 避免将已停止合作或未完成准入的供应商继续用于新业务。 |
| 供应物料范围 | 若需要限制可采购物料，应在供应商建立后继续维护供应商物料关系。 |

## 一项供应商如何进入业务

```mermaid
flowchart LR
    A["确认合作主体与编码口径"] --> B["新增供应商主数据"]
    B --> C["补充联系人、属性与可用状态"]
    C --> D["按需维护供应商物料"]
    D --> E["采购与到货业务选择供应商"]
    E --> F["发生变更时评估引用后再维护或停用"]
```

供应商创建完成不代表可立即用于所有采购场景。物料匹配、采购订单状态、到货规则和权限由对应业务页面控制；本页只负责提供可靠的主体资料和可用状态。

    以“华东紧固件有限公司”为例，展示供应商代码、简称、联系人、币种、可用状态，以及在供应商物料和采购收货中的使用方式。

!!! example "写实示例：给定配置 → 期望行为"
    **给定：** 新建供应商 `V-1001`（华东紧固件），代码/简称齐全，是否可用 = 是；尚未维护任何供应商物料。
    **期望：**

    1. 供应商列表与详情能按代码/简称找到该主体，且显示为可用。
    2. 采购/收货选择器中，在「仅可选可用供应商」通例下应能选到 `V-1001`（组织/权限裁剪 ❓）。
    3. 若业务要求必须带出供货物料关系，则仅有主体仍可能无法完成明细匹配——需继续维护[供应商物料](02-供应商物料.md)。
    4. 将是否可用改为否后，新单原则上不应再选到该供应商；过滤时点 ❓ 待确认。在途单据是否仍能展示历史供应方，以业务页为准。

## 关键字段业务角色

下表只列影响主体识别与可用性的关键项；完整语义见[维护与查询参考](03-供应商-维护与查询参考.md)。写法约定见[页面数据字典规范](../../02-业务模型/04-页面数据字典规范.md)。下游「仅可选可用供应商」通例见[通用选择器过滤惯例](../../02-业务模型/12-通用选择器过滤惯例.md)；停用后过滤时点本页仍标 `❓`。

| 字段/配置点 | 在系统中的作用 | 关键行为要点（取值/范围/联动/门禁） | 维护或操作时要警惕什么 |
| --- | --- | --- | --- |
| 供应商代码 | 主体业务识别号 | 新增必填、查重；更新保留原代码 | 勿为规避重复加临时后缀；已被采购/收货引用后勿改码 |
| 简称 / 名称 | 日常识别 | 简称新增必填；更新保留原简称 | 简称应采购/仓库/财务可共用 |
| 类型 / 币种 | 属性分类与结算口径 | 字典选择 | 不确定时先确认采购口径 |
| 税率 | 税务属性 | 页面限制 0～1、最多 6 位小数 | 错填影响结算理解，不自动改单据 |
| 是否可用 / 有效期 | 生命周期 | 新增可用必填；停用后选择器过滤时点 ❓ | 停用前查在途采购、收货与供应商物料 |
| 联系人邮件等 | 协同联系 | 导入时邮件必填，与页面口径可能不同 | 勿假设页面与导入必填完全一致 |

## 关键维护与变更

| 维护点 | 当前可确认的业务规则 | 使用建议 |
| --- | --- | --- |
| 代码 | 新增时必须填写，服务层会检查是否与已有供应商代码重复。 | 代码应稳定；不要为规避重复而添加临时后缀。 |
| 简称 | 新增时必须填写。 | 使用采购、仓库和财务能够共同识别的简称。 |
| 名称与联系信息 | 用于日常识别和联系。 | 变更前确认是否影响正在进行的采购或到货协同。 |
| 类型、币种、税率与有效期 | 用于表达供应商属性及其适用时间。 | 不确定口径时先向采购或主数据责任人确认。 |
| 代码与简称修改 | 当前更新逻辑会保留原代码和原简称。 | 若确需变更身份标识，应走主数据变更方案，不应绕过页面或接口。 |
| 可用/停用 | 可用性是供应商生命周期的重要控制信息。 | 停用前先确认未完成采购、到货和供应商物料关系；实际选择器过滤时点需测试验证。 |
| 删除 | 当前实现未能证明会拦截全部下游引用。 | 已被业务使用的供应商不建议删除，优先评估停用或归档。 |

### 可用性如何起作用

供应商页没有独立策略配置。对下游「能不能选到」主要靠本页可用性与有效期，以及各业务选择器通例：

| 维护点 | 起作用的方式 | 排障时先看 |
| --- | --- | --- |
| 是否可用 | 停用后原则上不应再被新单选中；过滤时点 ❓ | 是否已停用 |
| 生效/失效时间 | 表达适用窗口；是否自动拦截选择器 ❓ | 是否落在有效期内 |
| 供应商物料关系 | 主体可用仍可能因缺少关系而匹配失败 | 是否已建可用「供应商 + 物料」 |

现场说「选不到供应商」：先看本页是否可用与有效期，再查权限与业务页条件；若能选主体但物料对不上，转到[供应商物料](02-供应商物料.md)。

### 建议验证点

- 新建可用供应商后，采购相关选择器能否选到（有无权限裁剪）。
- 停用后各选择器是否立即过滤（有无时点差异）。
- 未维护供应商物料时，采购明细匹配/带入表现（强制还是仅提示 ❓）。
- 导入缺代码/简称/联系人邮件时是否形成可识别错误回执。

## 查询、详情与联查

日常查询建议先用供应商代码或名称定位，再用类型、币种、生效区间等条件缩小范围。详情页应优先让读者判断“是谁、是否可用、如何联系、供应什么、在哪些业务中被引用”。

### 详情分组与快速跳转

| 详情分组 | 应回答的问题 | 建议联查 |
| --- | --- | --- |
| 基本身份与状态 | 供应商是谁，当前是否可用。 | 供应商物料。 |
| 联系与结算属性 | 应联系谁，采用什么币种、税率或期限信息。 | 采购业务中的供应来源。 |
| 生命周期与审计 | 何时生效、何时失效、何时被维护。 | 变更记录（后续补充）。 |
| 业务引用 | 该供应商供应什么物料、是否已用于采购或到货。 | 供应商物料、采购订单、采购收货。 |

## 批量维护

供应商导入适合初始建档或按统一口径批量维护。导入前必须确认代码、简称和联系人邮件是否完整，并确认本次仅新增、仅更新还是允许两者同时处理；不要用导入绕过供应商准入和变更评估。

模板列、必填项、错误回执和处理方式见[供应商-维护与查询参考](03-供应商-维护与查询参考.md)。

## 常见问题与处理

| 情况 | 建议处理 |
| --- | --- |
| 新增提示代码重复 | 先查询既有供应商，确认是否应维护原资料；不要建立同一主体的重复代码。 |
| 业务中找不到供应商 | **排障入口：** ① 本页是否可用、是否在有效期内；② 当前用户是否有可见范围；③ 是否其实是「有供应商但缺供应商物料」——改查[供应商物料](02-供应商物料.md)；④ 业务页自身是否还有来源单据/状态条件。 |
| 能选供应商但物料带不出 | 转到供应商物料：查「供应商 + 物料」是否存在且可用，以及月台/换算是否齐全。 |
| 需要修改代码或简称 | 不要直接改写；先确认引用范围并按主数据变更流程处理。 |
| 导入出现错误行 | 按错误回执逐行修正代码、简称、邮件或重复数据，再重新导入。 |
| 已停止合作 | 先检查未完成业务和引用，再执行停用或归档；不要直接删除。 |

## 当前限制与待确认事项

- 供应商停用后在采购、收货等选择器中的实际过滤时点尚需测试环境验证；
- 删除是否会拦截采购、收货、退货、供应商物料等全部引用，尚未形成端到端测试结论；
- 供应商详情的实际页签、跳转过滤条件和变更日志入口需要用测试环境截图确认；
- 前端按钮可见性不能单独证明后端接口权限，动作权限需在 System/RBAC 轮次验证。
